# Neo-MoFox 三层架构总览：严格的职责边界设计

> **架构哲学**：三层分离，职责清晰，依赖单向——确保系统的可维护性、可扩展性和可测试性。

---

## 一、三层架构的设计哲学

### 1.1 传统框架的架构混乱

**问题现象**：多数AI聊天框架（Rasa、LangChain）存在架构混乱：

```
传统框架结构（混乱）：
┌─────────────────────────────┐
│   业务逻辑 + 技术基础设施     │  混在一起
│   + 平台适配 + 配置管理      │  无法分离
│   + 持久化 + LLM调用         │  耦合严重
└─────────────────────────────┘
```

**后果**：
- **不可维护**：修改一个功能可能影响多个模块
- **不可扩展**：增加新平台需修改核心代码
- **不可测试**：无法独立测试单个模块
- **不可复用**：技术基础设施无法在其他项目中使用

### 1.2 Neo-MoFox 的三层分离原则

**架构设计原则**：

1. **职责分离**：每层有明确的职责边界，不越界
2. **依赖单向**：只能从上到下依赖（app → core → kernel），禁止反向依赖
3. **平台无关**：kernel层不包含任何业务逻辑，core层不包含任何平台代码
4. **可测试性**：每层可独立测试，mock下层依赖

**三层架构**：

```
┌──────────────────────────────────────────┐
│         app（应用层）                      │
│   职责：组装 kernel+core，提供运行时       │
│   内容：Bot 运行时、插件系统、入口点        │
│                                          │
│   依赖：core + kernel                     │
│   禁止：被 core 或 kernel 依赖            │
└────────────────┬─────────────────────────┘
                 │ ↓ 单向依赖
                 │
┌────────────────┴─────────────────────────┐
│         core（领域层）                     │
│   职责：实现领域逻辑、插件组件              │
│   内容：组件基类、管理器、提示词、消息模型   │
│                                          │
│   依赖：kernel                            │
│   禁止：依赖 app 或包含平台代码             │
└────────────────┬─────────────────────────┘
                 │ ↓ 单向依赖
                 │
┌────────────────┴─────────────────────────┐
│         kernel（基础设施层）               │
│   职责：提供技术基础设施，无业务逻辑        │
│   内容：数据库、向量库、调度器、LLM、配置    │
│                                          │
│   依赖：外部库                             │
│   禁止：依赖 core/app 或包含业务逻辑        │
└──────────────────────────────────────────┘
```

### 1.3 职责边界定义

| 层级 | 职责范围 | 禁止内容 | 依赖方向 |
|-----|---------|---------|---------|
| **app** | 应用组装、运行时、插件加载 | 技术基础设施、业务逻辑实现 | → core + kernel |
| **core** | 领域逻辑、插件组件、管理器 | 平台代码、技术基础设施 | → kernel |
| **kernel** | 技术基础设施、工具库 | 业务逻辑、领域概念 | → 外部库 |

**关键约束**：

```python
# ✅ 正确依赖
from src.app import Bot
from src.core import PluginManager
from src.kernel import LLMRequest

# ❌ 错误依赖（禁止反向）
# kernel/core 依赖 app（禁止）
# core/kernel 依赖 plugins（禁止）
```

### 2.2 Kernel 层特性

#### 特性 1：平台无关

**代码不包含任何平台逻辑**：

```python
# kernel/db/crud.py - 纯数据库操作
class CRUDBase:
    async def get_by(self, **filters) -> Model:
        """通用 CRUD 查询，无业务逻辑。"""
        # 纯 SQLAlchemy 操作
        query = select(self.model).filter_by(**filters)
        return await self.session.execute(query)
```

**没有QQ/Discord/Telegram相关代码**，平台适配在 app层的 Adapter 中。

#### 特性 2：业务逻辑隔离

**不包含领域概念**：

```python
# kernel/llm/request.py - 纯LLM调用
class LLMRequest:
    async def send(self) -> LLMResponse:
        """LLM请求发送，无"对话"、"消息"等业务概念。"""
        # 纯 API 调用，payload 只是抽象数据结构
        return await self._client.call(self.payloads)
```

**"对话"、"记忆"、"消息"等概念在 core层**。

#### 特性 3：工具库性质

**可独立使用**：

```python
# kernel 层可单独在其他项目中使用
from src.kernel.config import ConfigBase
from src.kernel.logger import get_logger

# 无需 core/app，kernel 自身就是完整工具库
```

```python
# kernel/llm/client.py
class LLMClient:
    """统一LLM客户端，支持OpenAI、Claude、本地模型。"""
    
    def __init__(self, model_set: ModelSet):
        """
        Args:
            model_set: 配置中的模型集合（从 config/model.toml 加载）
        """
        self._provider = model_set.provider  # openai/claude/local
        self._client = self._create_client()
    
    async def call(self, payloads: list[LLMPayload]) -> LLMResponse:
        """统一调用接口，屏蔽厂商差异。"""
        if self._provider == "openai":
            return await self._openai_call(payloads)
        elif self._provider == "claude":
            return await self._claude_call(payloads)
```

```python
# kernel/config/base.py
class ConfigBase(BaseModel):
    """配置基类，支持TOML加载、热重载、类型验证。"""
    
    @classmethod
    def load(cls, path: str) -> Self:
        """从TOML文件加载配置。"""
        data = toml.load(path)
        return cls.model_validate(data)
    
    def reload(self, path: str) -> bool:
        """热重载配置（文件变化时自动更新）。"""
        new_config = self.load(path)
        if new_config != self:
            self.__dict__.update(new_config.__dict__)
            return True
        return False
```

#### 2.3.3 并发管理（kernel/concurrency）

**TaskGroup + WatchDog**：

```python
# kernel/concurrency/task_manager.py
class TaskManager:
    """统一任务管理器。"""
    
    async def group(
        self,
        name: str,
        timeout: float | None = None,
        cancel_on_error: bool = True,
    ) -> TaskGroup:
        """创建任务组（自动清理、超时控制）。"""
        return TaskGroup(
            name=name,
            timeout=timeout,
            cancel_on_error=cancel_on_error,
        )
    
    def create_task(
        self,
        func: Callable,
        name: str,
        daemon: bool = False,
    ) -> TaskInfo:
        """创建后台任务（自动监控）。"""
        task = asyncio.create_task(func())
        self._watchdog.register(task, name, daemon)
        return TaskInfo(task_id=task.get_name(), name=name)
```

**WatchDog 监控**：

```python
# kernel/concurrency/watchdog.py
class WatchDog:
    """任务监控器（检测僵尸任务、超时任务）。"""
    
    async def check_health(self) -> dict[str, Any]:
        """检查所有任务健康状态。"""
        return {
            "alive_tasks": len(self._alive_tasks),
            "zombie_tasks": len(self._detect_zombies()),
```

#### 3.3.2 管理器（core/components/managers）

**PluginManager**：

```python
# core/managers/plugin_manager.py
class PluginManager:
    """插件加载与管理。"""
    
    async def load_plugin(self, path: str | Path) -> PluginInfo:
        """加载插件（支持folder/zip/.mfp）。"""
        # 读取 manifest.json
        manifest = self._read_manifest(path)
        
        # 导入插件模块
        plugin_module = import_module(path)
        
        # 注册插件类
        plugin_class = self._find_plugin_class(plugin_module, manifest.name)
        self._registry.register_plugin(plugin_class)
        
        # 注册所有组件
        components = plugin_class().get_components()
        for comp in components:
            self._registry.register_component(comp, plugin_class)
```

**ToolManager**：

```python
# core/managers/tool_manager.py
class ToolManager:
    """工具管理与执行。"""
    
    def generate_schema(self, tools: list[BaseTool]) -> list[dict]:
        """生成LLM工具schema。"""
        schemas = []
        for tool in tools:
            schema = {
                "name": tool.tool_name,
                "description": tool.tool_description,
                "parameters": self._extract_params(tool.execute),
            }
            schemas.append(schema)
        return schemas
    
    async def execute_tool(self, tool_name: str, args: dict) -> tuple[bool, Any]:
        """执行工具调用。"""
        tool_class = self._registry.get_tool(tool_name)
        tool_instance = tool_class(plugin=self._plugin)
        return await tool_instance.execute(**args)
```

#### 3.3.3 Prompt 系统（core/prompt）

**PromptTemplate**：

```python
# core/prompt/template.py
class PromptTemplate:
    """Prompt模板引擎。"""
    
    def __init__(self, name: str, template: str, values: dict):
        self.name = name
        self.template = template
        self.values = values
    
    def build(self) -> str:
        """构建完整Prompt（支持变量注入、事件订阅）。"""
        # 发布 on_prompt_build 事件

            "name": self.name,
            "template": self.template,
            "values": self.values,

        # 变量替换
        result = self.template.format(**self.values)
        return result
```

**SystemReminder**：

```python
# core/prompt/system_reminder.py
class SystemReminderStore:
    """System Reminder存储（注入到LLM prompt）。"""
    
    def set(self, bucket: str, name: str, content: str):
        """设置reminder。"""
        self._store[bucket][name] = content
    
    def get_all(self, bucket: str) -> list[str]:
        """获取bucket内所有reminder。"""
        return list(self._store[bucket].values())
    
    def inject_to_payload(self, bucket: str) -> list[LLMPayload]:
        """注入reminder到LLM payload。"""
        reminders = self.get_all(bucket)
        return [
            LLMPayload(ROLE.SYSTEM, Text(f"<system_reminder>{r}</system_reminder>"))
            for r in reminders
        ]
```

### 4.3 App 层关键模块

#### 4.3.1 Bot 运行时（app/runtime/bot.py）

```python
class Bot:
    """Bot 主运行时。"""
    
    async def start(self):
        """启动Bot。"""
        # 加载配置
        await self._load_configs()
        
        # 加载插件
        await self.plugin_manager.load_all()
        
        # 启动适配器（连接平台）
        adapters = self.plugin_manager.get_all_adapters()
        for adapter in adapters:
            await adapter.start()
        
        # 启动主循环
        await self._main_loop()
    
    async def _main_loop(self):
        """主事件循环。"""
        while self.running:
            # 接收平台消息
            message = await self._receive_message()
            
            # 找到对应的 Chatter
            chatter = self.chatter_manager.get_chatter(message.stream_id)
            
            # 执行对话流程
            await chatter.execute()
```

## 五、依赖方向与架构约束

### 5.1 单向依赖规则

**依赖图**：

```
app → core → kernel → external_libs
 ↓     ↓      ↓
禁止反向依赖（kernel → core, core → app）
```

**代码检查**：

```python
# ✅ 正确依赖
# app/runtime/bot.py
from src.core.managers import PluginManager  # app → core ✅

# core/components/base/action.py

# ❌ 错误依赖（禁止反向）
```

### 5.2 架构边界检查

**自动化检查脚本**：

```python
# scripts/check_architecture.py
"""检查三层架构边界。"""

import ast
import pathlib

def check_imports():
    """检查所有import是否符合架构规则。"""
    errors = []
    
    for file in pathlib.Path("src").glob("**/*.py"):
        module = ast.parse(file.read_text())
        
        for node in ast.walk(module):
            if isinstance(node, ast.ImportFrom):
                source = file.relative_to("src").parts[0]  # kernel/core/app
                target = node.module.split(".")[0]        # kernel/core/app
                
                # 检查反向依赖
                if source == "kernel" and target in ("core", "app"):
                    errors.append(f"{file}: kernel → {target} (禁止反向)")
                elif source == "core" and target == "app":
                    errors.append(f"{file}: core → app (禁止反向)")
    
    return errors
```

### 5.3 架构约束的好处

| 维度 | 无架构约束 | 有架构约束 |
|-----|-----------|-----------|
| **可维护性** | 修改一处影响多处 | 修改局限在一层 |
| **可测试性** | 无法独立测试 | 每层可mock依赖测试 |
| **可扩展性** | 新功能需修改核心 | 新功能通过插件扩展 |
| **可复用性** | kernel无法独立使用 | kernel可在其他项目使用 |

## 六、与微服务架构的对比

### 6.1 微服务 vs 三层架构

| 维度 | 微服务架构 | Neo-MoFox三层架构 |
|-----|-----------|------------------|
| **粒度** | 服务级（独立进程） | 层级（同一进程） |
| **通信** | RPC/HTTP（网络） | 函数调用（内存） |
| **隔离** | 进程隔离（强） | 层级隔离（软） |
| **部署** | 独立部署 | 统一部署 |
| **性能** | 网络开销 | 内存调用 |
| **复杂度** | 高（分布式） | 低（单体） |

**Neo-MoFox选择三层架构的原因**：

- **单进程性能**：AI聊天系统需要高频调用LLM，网络开销不可接受
- **状态连续性**：Life Engine心跳需要内存级延迟，不适合分布式
- **部署简单**：单体应用部署复杂度低，适合个人开发者

### 6.2 三层架构的适用场景

**适用**：
- 单体应用
- 高性能要求（内存调用）
- 状态连续性要求
- 个人开发/小团队

**不适用**：
- 大规模分布式系统
- 需要独立部署不同模块
- 团队超过100人

## 七、架构演进路径

### 7.1 当前架构（单体三层）

```
src/
├── kernel/  # 基础设施
├── core/    # 领域逻辑
├── app/     # 应用组装
└── plugins/ # 插件（app层扩展）
```

### 7.2 演进方向（可选）

**方向 1：微内核架构（未来）**

```
# kernel变成真正的微内核
src/kernel/  → 独立进程（基础设施服务）
src/core/    → 插件进程（领域逻辑）
src/app/     → 协调器进程

# RPC通信
kernel.llm.call() → RPC → llm_server.call()
```

**触发条件**：
- 需要独立部署LLM服务
- kernel被多个项目共享
- 团队规模增长

**方向 2：插件隔离（未来）**

```
# 每个插件独立进程
plugins/life_engine/ → 独立进程
plugins/default_chatter/ → 独立进程

# 进程间通信
life_engine → nucleus_tell_dfc → RPC → default_chatter
```

**触发条件**：
- 插件崩溃不影响主系统
- 插件可独立重启
- 插件沙箱隔离（安全）

## 八、总结

### 8.1 三层架构的核心价值

| 维度 | 传统混乱架构 | Neo-MoFox三层架构 |
|-----|-------------|------------------|
| **职责清晰** | 模糊，耦合严重 | 明确，三层分离 |
| **依赖管理** | 双向依赖，混乱 | 单向依赖，清晰 |
| **可测试性** | 无法独立测试 | 每层可mock测试 |
| **可维护性** | 修改影响全局 | 修改局限一层 |
| **可扩展性** | 修改核心代码 | 插件扩展 |
| **可复用性** | 无法复用 | kernel可独立使用 |

### 8.2 架构约束的代价与收益

**代价**：
- 需严格遵守依赖规则
- 不能随意跨层调用
- 需额外抽象层

**收益**：
- 代码长期可维护
- 重构成本降低
- 测试覆盖率可达100%
- 新功能通过插件扩展（不修改核心）

### 8.3 架构哲学总结

Neo-MoFox的三层架构不是复杂的设计，而是**严格的约束**：

- **约束1**：单向依赖（app → core → kernel）
- **约束2**：职责边界（kernel无业务、core无平台、app组装）
- **约束3**：平台无关（kernel/core不含平台代码）

这些约束确保系统长期健康发展，避免架构腐化。
