# 第 3 章 · 三条工程约束

> *"概念可以解释方向，但真正能被课程报告检查的是约束：系统有没有心跳、有没有状态、有没有本地更新、有没有可观察的闭环。"*

---

本章保留 Neo-MoFox 的三条设计原则，但不再展开成完整哲学论证。对本报告而言，它们首先是工程约束，用来回答三个问题：

| 原则 | 工程问题 | Neo-MoFox 的对应机制 |
|---|---|---|
| 连续性 | 外部输入之间系统是否仍在演化 | Life Engine 心跳、SNN tick、调质 ODE、状态持久化 |
| 自下而上学习 | 系统是否能从交互中本地更新 | 软 STDP、Hebbian 记忆边强化、习惯 streak、记忆衰减 |
| 系统涌现智能 | 智能是否来自多个子系统协作 | LLM + LifeChatter + Life Engine + Memory + Dream + SNN + Neuromod + ThoughtStream |

## 3.1 连续性：输入间隙不能是空白

连续性不是说系统每一秒都必须输出文本，而是说系统在没有用户输入时仍然有内部状态在推进。对 Neo-MoFox 来说，这个约束落在四个具体机制上：

- `LifeEngineService` 以心跳节律采样事件、推进状态并写入持久化文件；
- SNN 在独立 tick 中执行真实 step 或 decay-only，保留短时间尺度的状态余韵；
- 神经调质变量按 ODE 方式回归、衰减或受刺激项驱动；
- 关键状态写入 `data/life_engine_workspace/`，使重启不是从空白开始。

因此，“连续存在”在本文中不是形而上判断，而是可检查的运行时事实：如果关闭心跳、SNN tick 和状态写盘，系统就会退化成普通的调用式聊天程序。

## 3.2 自下而上学习：运行过程就是更新过程

本报告不把“学习”限定为离线训练。对一个长期运行的 Agent 来说，更重要的是部署后的本地更新：

- SNN 使用软 STDP，让前后活动度和奖赏信号改变突触权重；
- 记忆图使用 Hebbian 强化，让共同激活的节点边权升高；
- 记忆衰减和访问强化让长期上下文不是静态仓库；
- 习惯 streak 把重复行为转化为可被调质层读取的倾向。

这与简单 if-else 规则不同。规则可以模拟行为，但不会让系统结构随真实交互改变。Neo-MoFox 仍保留一些规则化遗留路径，这些会在第 13 章作为工程债务说明；但本报告强调的是已落地的本地可塑性机制。

## 3.3 系统涌现智能：不是单一模型包打天下

Neo-MoFox 的基本判断是：LLM 很强，但它不应该承担系统中的全部角色。当前系统把能力拆成多层：

- LLM 负责语言、推理和高层整合；
- LifeChatter 负责对话上下文、行动边界和工具调用控制；
- Life Engine 负责后台心跳、事件流和内部状态推进；
- Memory / Dream / ThoughtStream 提供长期上下文、离线巩固和中期注意力；
- SNN / Neuromod 提供快速事件响应、慢变量调节和本地可塑性。

这些模块单独看都不完整，组合后才形成“感知-状态-决策-反馈”闭环。第 4 章将从工程框架说明这些模块如何挂载，第 5–10 章再分别展开关键子系统。

## 3.4 与智能感知及无人系统的对应

| 课程概念 | Neo-MoFox 中的工程对应 |
|---|---|
| 持续感知 | 消息、工具调用、静默时长、梦境残影和内部事件进入统一事件流 |
| 状态估计 | runtime snapshot 汇总调质、记忆、ThoughtStream、近期事件和内心独白 |
| 自主决策 | LifeChatter 在多轮状态机中决定回复、等待、调用工具或记录内心独白 |
| 在线适应 | 软 STDP、记忆边强化、习惯 streak 和调质状态随交互变化 |
| 故障恢复 | JSON / SQLite / 索引持久化支撑重启恢复 |

这种对应不是说 Neo-MoFox 等同于物理无人系统，而是说明无人系统的闭环思想可以迁移到软件 Agent 中。

## 3.5 可证伪性

三条原则仍然需要实证约束：

| 原则 | 可检查方式 |
|---|---|
| 连续性 | 关闭心跳或状态持久化，观察输入间隙后的行为差异是否消失 |
| 自下而上学习 | 冻结 SNN 权重和记忆边，观察长期偏好与检索路径是否停止演化 |
| 系统涌现 | 单独保留 LLM，切除 Life Engine 子系统，观察状态感、主动性和长期一致性是否下降 |

目前项目具备部分消融开关和日志基础，但尚缺乏长周期实测数据。这个不足会在第 13 章明确列出。

## 3.6 小结

本章把概念压缩为工程约束。后续章节不再反复解释“为什么要像生命”，而是说明 Neo-MoFox 如何通过真实框架、真实插件、真实状态文件和真实运行流程，把这些约束落到可检查的代码结构上。
