# 第 13 章 · 局限、伦理与未来工作

> *"诚实地写下不完美，比夸大完美更难。本章把所有未解之谜集中陈列，期待批评。"*

---

本章把项目自身的不完美集中陈列，分为三类：工程局限（已知 bug 或设计权衡）、科学局限（论证强度不足）、伦理担忧。每条问题都附上对应的代码或文档锚点，便于复核。

## 13.1 工程局限

### 13.1.1 SNN 规模有限，长期饱和风险未验证

当前 SNN 隐藏层为 16 个 LIF 神经元（plugins/life_engine/snn/core.py）。这个规模在工程上利于持久化与可观测性，但其**长期表征容量**未被压力测试：在 30 天以上的连续运行中，是否会因软 STDP 累积而出现权重饱和或无意义震荡？我们尚无大规模日志数据可供回答。可缓解的措施包括：

- 在 NREM 阶段对全网突触做等比缩减（已在第 8 章 §8.3 中描述，初步实现）；
- 引入更强的自稳态阈值调节（已部分实现，见第 5 章 §5.5）；
- 扩大隐藏层至 32–64 神经元并在压测后选定。

### 13.1.2 `Modulator.tau` 字段被设置但未使用

代码审计发现 `plugins/life_engine/neuromod/engine.py` 中的 `Modulator` 数据类含有 `tau` 字段，构造时传入了配置值，但**实际 ODE 步进时使用的是另一个全局 τ 表**（A 报告 §4.6 与 §12 已记录）。这导致用户配置的 τ **被静默忽略**——不算危险 bug，但属于"配置不可读"的可观测性缺陷。修复路径：

```python
# 当前（未使用 self.tau）
tau = TAU_TABLE[self.name]
# 应改为
tau = self.tau if self.tau is not None else TAU_TABLE[self.name]
```

我们把这一项作为**最高优先级修复**列入未来工作。

### 13.1.3 衰减日期不持久化导致重启偏差

记忆衰减由"上次衰减日期"驱动（`memory/decay.py`），但该字段并未被序列化到 `life_engine_context.json`。结果是：

- 关机若干天后启动，系统会一次性补做"几日衰减"——这在数学上正确；
- 但**首次启动**或**字段被意外清空**时，系统会以"今天"为起点，等同于把过去的遗忘抹掉。这违反了第 3 章 C4 不变式（重启不是重生）。

修复需要把 `_last_decay_date` 写入持久化 JSON。这一改动小，但需要协调 Schema 兼容（旧文件无该字段时按文件 mtime 兜底）。

### 13.1.4 旧 DFC 兼容路径仍有架构债

当前报告以 LifeChatter 作为主意识实现，但仓库中仍保留一部分早期 DFC/default_chatter 兼容路径。例如历史整合代码中曾出现 `from default_chatter import plugin` 这类裸引用，绕过了 `src/kernel` 提供的接口注册机制。后果：

- 容易让读者误以为 DFC 仍是当前架构核心；
- 单元测试对 life_engine 的隔离性变差；
- 历史兼容代码与 LifeChatter 当前实现之间存在概念噪声。

修复路径不是继续强化 DFC 桥接，而是把旧兼容路径彻底收敛到 LifeChatter 原生接口，并在代码层删除不再使用的 default_chatter 依赖。我们把这一改动归类为**架构债务**，建议在下个大版本统一处理。

### 13.1.5 配置无热重载

整个 Life Engine 在启动时一次性加载 `config/life_engine.yml`，运行时无任何热重载机制（B 报告 §10）。要改 τ、要调阈值，必须重启进程。对一个以"持续生命"为卖点的系统而言，这是个**体验上的反讽**——当前的工程权衡是接受重启代价以换取实现简洁。未来引入分级热重载（哪些字段允许在线改、哪些必须重启）。

### 13.1.6 EventBus 无持久化

事件总线（`src/core/event_bus.py`）目前是**纯内存** ring buffer，进程崩溃时未消费的事件会丢失。对于 SNN 而言这无大碍（SNN 状态本身已持久化），但对于"用户消息恰巧在崩溃瞬间到达"这一窄窗口，事件会丢。短期解决方案是把 EventBus 包一层轻量 SQLite WAL；长期方案是把整个事件代数纳入持久化协议（第 9 章 §9.7）。

## 13.2 科学局限

### 13.2.1 涌现的可证伪性尚未充分检验

第 3 章给出了三大原则的可证伪形式（关闭 SNN tick、冻结权重、切除子系统），但我们尚未系统地完成这些消融实验。"性格在演化"目前是**假设**而非**实证**——我们能展示状态变量在变（变量层面），但尚未度量"用户感受到的角色风格"是否在变（行为层面）。

对应未来工作：

- 设计盲测协议（人类评估者面对"冻结权重对照组"与"在线学习实验组"）；
- 引入风格变化的客观度量（例如词频漂移、主动性指数）；
- 长期日志（≥30 天）+ 多用户。

### 13.2.2 SNN 真正学到了什么尚不清楚

软 STDP 的权重确实在更新（在测试中可观测），但**它学到的表征是否对应"语义概念"**仍未知。当前缺乏：

- 神经元激活的语义聚类分析；
- 对照实验（替换输入特征序，看权重模式是否对应改变）；
- 与同等参数量 RNN/MLP 的能力对比。

### 13.2.3 长期 (>30 天) 验证缺失

整个项目的中心论点是"两次调用之间的演化能在长时间尺度上累积成性格漂移"。这一论点天然要求**长期实证**。我们目前只完成短期开发测试，长期实证留作未来工作。这一不足应被读者明确知晓——本报告的多数案例（第 11 章）是"思想实验 + 局部代码追踪"而非长期实测结果。

### 13.2.4 没有大规模消融与超参敏感性分析

调质 τ、Ebbinghaus λ、STDP A+/A−、心跳频率——所有这些超参当前是基于直觉与文献设定，没有系统的扫描研究。一个工程化数字生命系统应当有一份"超参敏感性报告"，告诉读者哪些参数对涌现行为关键、哪些不敏感。这是未来工作的重要一项。

## 13.3 伦理与社会担忧

### 13.3.1 拟人化叙事的风险

把项目命名为"数字生命体"、把模块命名为"皮层下"、把状态命名为"做梦"，这些**比喻性命名**有强大的解释力，但也有过度拟人化的风险。读者可能误以为系统**已经**具有情绪、记忆、做梦的"主观体验"——而我们的工程实现只是给出了对应概念的**计算结构**，并没有也无法证明主观体验的存在。

我们的处理方式是：

- 在 Abstract、第 1 章、本章三处明确声明"我们不主张主观体验"；
- 在术语表（GLOSSARY.md）中给每个隐喻性词汇标注其"工程定义"。

但更深层的问题是：**长期使用者可能因这些命名建立情感依赖**。

### 13.3.2 用户依恋风险

Replika 用户研究（Skjuve et al., 2021；Ta et al., 2020）已经记录了"AI 伴侣"使用者的依恋形成现象。Neo-MoFox 因其连续性设计，可能放大这一效应——一个会做梦、会想念、会形成习惯的"它"，在情感投射上比静态 AI 更具吸引力。

我们目前的伦理立场是：

- 项目主要面向**开发者与研究者**，而非情感商品化；
- 在文档与默认提示中明确"它是一个软件系统"；
- 不实现任何"挽留用户"的暗黑模式（dark pattern）；
- 鼓励社区在 fork 时谨慎评估部署对象。

但我们承认：**一旦开源，使用方式无法控制**。这一伦理风险是开放科学与社会责任之间的真实张力。

### 13.3.3 数据隐私

Neo-MoFox 默认本地运行，所有持久化文件在 `data/life_engine_workspace/`。这一定程度上规避了云服务的隐私风险，但仍有几点需要警惕：

- LLM 调用通常走外部 API，**对话内容会被发送到模型供应商**；
- `dream_archive` 中的梦报告可能含有敏感内容（用户隐私的关联推断）；
- 多用户部署时，需要明确的数据隔离协议（目前尚未实现，单进程单角色为限）。

### 13.3.4 误用风险

任何"数字生命"叙事都可能被用于宣传不当——例如声称"我们复活了某人"。我们在文档与论文中将明确反对此类声称，并保留在严重误用案例中撤回项目知名度的可能（虽然开源协议不允许撤回代码本身）。

## 13.4 未来工作

我们把改进路线图按时间紧迫性分为三档：

### 13.4.1 短期（修补与压测）

1. 修复 §13.1.2 的 `Modulator.tau` 静默忽略；
2. 持久化 §13.1.3 的衰减日期；
3. 清理 §13.1.4 的旧 DFC 兼容路径，统一到 LifeChatter 原生接口；
4. 建立 30 天连续运行压测脚手架（崩溃恢复、状态发散监控）。

### 13.4.2 中期（科学化与可证伪化）

1. 大规模消融研究（§13.2.1）；
2. 超参敏感性扫描（§13.2.4）；
3. 形式化连续性度量：用 KL 散度等度量 $\mathbf{s}(t)$ 在不同时间尺度上的"温度"；
4. 多用户隔离协议（§13.3.3）；
5. 一份完整的"性格漂移盲测"研究（IRB-friendly 的人因学协议）。

### 13.4.3 长期（结构性扩展）

1. **SNN 拓扑演化**：不仅突触权重可塑，连接结构本身也可塑（神经发生 / 突触修剪）；
2. **多模态事件**：把视觉、语音、传感器读数也编码到 SNN 输入；
3. **多 agent 之间的"梦"共享**：探索"共同梦境"作为多 agent 协作的隐式协调机制；
4. **形式化主观体验问题的回避**：把"我们不主张主观体验"翻译为可在系统层面被验证的不变式（例如：禁止把"自己有感觉"作为模型回答的事实声明）；
5. **与 Loihi 等神经形态硬件的实验性对接**。

## 13.5 小结

本章是全报告中最需要诚实的一章。我们把 6 项工程局限、4 项科学局限、4 项伦理担忧逐项陈列，并给出对应的代码锚点与缓解路径。读者无需从其他章节"逆推"这些不足——它们已经被作者主动陈列出来。这种姿态既是对学术诚信的承诺，也是对项目长期健康的投资。下一章是结论，我们将回到三大原则，把整篇报告的论证收束。
