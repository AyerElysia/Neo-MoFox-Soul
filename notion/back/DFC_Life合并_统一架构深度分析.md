# DFC 与 Life 合并：统一架构深度分析

> 作者：Copilot (Claude Opus 4.6)  
> 日期：2026-04-17  
> 背景：用户多次反馈 DFC 严重拖累 life_engine 性能，信息差无法弥补  

---

## 一、问题本质：为什么 DFC 拖累了 Life？

### 1.1 信息鸿沟——80-90% 的上下文在传输中丢失

| 维度 | life_engine 拥有 | DFC 能看到 | 信息保留率 |
|------|------------------|------------|-----------|
| 情绪状态 | SNN + 调质层完整向量 | 3 个离散标签（好奇/精力/满足） | ~5% |
| 事件流 | 300+ 事件，含时序/来源/类型 | 最近 2 条心跳独白（40 字摘要） | ~3% |
| 记忆网络 | STDP 激活 + 联想扩散 | 需主动调用工具才能检索 | 0%（被动） |
| 工具偏好 | 完整使用历史 | 最近 2 个高频工具名 | ~10% |
| TODO 系统 | 完整优先级/进度/上下文 | 需 consult_nucleus 主动查 | 0%（被动） |
| 梦境残留 | 完整 DreamTrace + 残留衰减 | 一行余韵文本 | ~5% |

**结论：DFC 本质上是在信息饥饿的状态下做决策。**

### 1.2 架构性瓶颈——不是桥不够宽，是桥本身就不该存在

我们尝试过的每一次"桥接优化"：

| 尝试 | 效果 | 为什么不够 |
|------|------|-----------|
| SystemReminder 注入 | 只在 runner init 读一次，10 分钟后就过期 | 架构限制：DFC runner 不刷新 |
| Life State 临时注入 | 每轮 MODEL_TURN 前注入 | 位于 payloads 末尾，破坏缓存 |
| consult_nucleus 工具 | DFC 主动查询 | 多花 1 次 LLM 调用 + 延迟 |
| nucleus_tell_dfc 主动传话 | 异步队列，不保证时效 | DFC 不一定在听 |
| retrieve_memory 智能检索 | 内部用 sub_agent | 又套了一层 LLM，延迟 0.8-2s |

**问题不在于桥不够宽，而在于：为什么要隔着一条河喊话，而不是直接站在同一个房间里？**

### 1.3 Token 浪费的量化

一次典型的 DFC 对话回合消耗：

```
System Prompt（人设+安全+工具说明）   ≈ 1500 tokens（固定）
Sub-agent 决策（群聊时）            ≈ 1000 tokens（每轮）
Life State 注入                    ≈  200 tokens（每轮，且多数过期）
Chat History                       ≈  500 tokens（累积）
User Message                       ≈  100 tokens
DFC Response                       ≈  300 tokens
───────────────────────────────────────────
单轮总计                           ≈ 3600 tokens
```

对比如果 life_engine 直接回话：

```
System Prompt（SOUL.md，已缓存）      ≈ 1200 tokens（100% 可缓存）
内部状态（原生访问，0 额外 token）    ≈    0 tokens
Chat Context（事件流中的消息）        ≈  400 tokens
Response                            ≈  300 tokens
───────────────────────────────────────────
单轮总计                            ≈ 1900 tokens（节省 47%）
```

## 二、Reverie 的启示：统一执行，提示词分化

### 2.1 Reverie 核心模式

Reverie 的设计理念可以用一句话概括：**不是"思考系统"和"对话系统"的合作，而是同一个系统的两种运行模式。**

```
                        ┌─────────────────┐
                        │   soul.md 身份锚 │
                        └────────┬────────┘
                                 │
                    ┌────────────┼────────────┐
                    │            │            │
             ┌──────▼──────┐  ┌─▼──────────┐ │
             │  Autonomous │  │   User Turn │ │
             │   Intent    │  │   Event     │ │
             └──────┬──────┘  └──────┬──────┘ │
                    │                │        │
                    └────────┬───────┘        │
                             │                │
                    ┌────────▼────────┐       │
                    │  TurnOrchestrator│       │
                    │  (统一执行引擎)   │       │
                    └────────┬────────┘       │
                             │                │
                    ┌────────▼────────┐       │
                    │  Shared Memory  │◄──────┘
                    │  (单一事实来源)  │
                    └─────────────────┘
```

**关键设计：**
1. **无独立聊天层** — 用户消息和自主思考走同一条执行管线
2. **提示词分化** — 不同模式仅改变 system prompt 的前言，不改变执行逻辑
3. **共享记忆池** — 思考和对话查询同一套记忆，不存在"聊天记忆"vs"认知记忆"的割裂
4. **soul.md 身份锚** — 所有模式共享同一份身份约束

### 2.2 这对我们意味着什么

| Reverie 概念 | 我们的对应 | 当前状态 | 改造方向 |
|-------------|-----------|---------|---------|
| Autonomous Turn | life_engine 心跳 | ✅ 已有 | 保持 |
| User Turn | DFC 对话 | ❌ 独立系统 | **合并进 life_engine** |
| TurnOrchestrator | 无 | ❌ 缺失 | **新建统一执行器** |
| soul.md | SOUL.md | ✅ 已有 | 保持 |
| Shared Memory | life_memory | ✅ 已有 | 保持（移除 DFC 侧重复） |
| SubconsciousPrompter | 心跳 prompt builder | ✅ 已有 | 扩展支持 chat 模式 |

## 三、统一架构设计

### 3.1 核心思想：Life 即 Chatter

```
                    ┌───────────────────────────┐
                    │      Neo-MoFox 框架        │
                    │  StreamLoopManager         │
                    │      ↓ tick                │
                    │  ChatterManager            │
                    │      ↓ select              │
                    │  LifeChatter ◄────────────── 不再是 DefaultChatter
                    │      ↓ execute()           │
                    └───────────┬───────────────┘
                                │
                    ┌───────────▼───────────────┐
                    │    LifeEngineService       │
                    │                           │
                    │  ┌─────────┐ ┌─────────┐  │
                    │  │ Heartbeat│ │  Chat   │  │
                    │  │  Mode   │ │  Mode   │  │
                    │  └────┬────┘ └────┬────┘  │
                    │       │           │       │
                    │  ┌────▼───────────▼────┐  │
                    │  │  Unified Executor   │  │
                    │  │  (统一执行引擎)      │  │
                    │  └────────┬────────────┘  │
                    │           │               │
                    │  ┌────────▼────────────┐  │
                    │  │  Shared Context     │  │
                    │  │  Event Stream       │  │
                    │  │  Memory / SNN       │  │
                    │  │  Tools / Neuromod   │  │
                    │  └────────────────────┘  │
                    └───────────────────────────┘
```

### 3.2 两种执行模式

#### 模式 A：心跳模式（Autonomous / Heartbeat）
- **触发**：定时器（每 N 秒）
- **输入**：事件流中的累积事件 + 内部状态
- **System Prompt**：SOUL.md + 内在认知框架
- **可用工具**：全部 25+ 工具（文件/TODO/记忆/网络/传话）
- **输出**：内部独白 + 工具调用
- **特点**：不直接对用户说话，思考为主

#### 模式 B：对话模式（Chat / Response）
- **触发**：收到用户消息
- **输入**：用户消息 + 相关事件流上下文 + 内部状态（原生访问）
- **System Prompt**：SOUL.md + 对话框架（含回复风格/安全准则）
- **可用工具**：send_text + pass_and_wait + 全部 life 工具
- **输出**：对用户的回复 + 可选工具调用
- **特点**：**天然拥有 life 的全部上下文**，无信息鸿沟

### 3.3 上下文管理：不污染，不浪费

**当前问题**：DFC 把整个聊天历史塞进 LLM 上下文，而 life 的事件流是另一套。

**统一方案**：

```
LLM Context (对话模式):
┌─────────────────────────────────────────┐
│ SYSTEM: SOUL.md + 对话框架              │  ← 100% 可缓存
│         + 安全准则 + 工具说明            │
├─────────────────────────────────────────┤
│ TOOL: 工具定义                          │  ← 100% 可缓存
├─────────────────────────────────────────┤
│ USER: 对话上下文                         │
│   【内在状态】(原生读取，0 额外开销)      │
│   【近期事件摘要】(从事件流提取)          │
│   【聊天记录】(从事件流中筛选消息类事件)   │
│   【新消息】当前用户消息                  │
├─────────────────────────────────────────┤
│ ASSISTANT: 回复                          │
└─────────────────────────────────────────┘
```

**关键设计：**
1. **事件流即历史** — 聊天消息只是事件流中的一种事件类型，不需要单独维护
2. **内部状态零开销** — 直接读取 self._inner_state，不需要序列化传输
3. **选择性注入** — 对话模式只注入与当前对话相关的事件，不是全部 300+ 事件
4. **System Prompt 100% 可缓存** — 不注入任何动态内容

### 3.4 Sub-agent 决策的简化

**现状**：DFC 用独立的 sub_agent LLM 调用来决定"要不要回复这条消息"（群聊场景）。消耗 ~1000 tokens/次。

**统一架构下**：
- **私聊**：直接回复（已有的行为）
- **群聊**：life_engine 本身已经能感知所有消息（事件流）。可以用简单规则（@提及 / 关键词 / SNN 激活度）做初筛，必要时才调用 LLM
- **进一步**：心跳模式中 life 已经在"想"这些消息了，如果她觉得需要回复，可以主动触发对话模式

## 四、为什么合并而不是继续优化桥接？

### 4.1 桥接方案的根本限制

```
桥接方案: life ──[serialize]──> 文本 ──[inject]──> DFC prompt ──[LLM]──> 回复
                      ↑                    ↑                      ↑
                  信息压缩损失         位置破坏缓存          两次 LLM 开销
                  
统一方案: life ──[直接访问 self._*]──> prompt builder ──[LLM]──> 回复
                      ↑                                    ↑
                  零信息损失                           一次 LLM 开销
```

### 4.2 从"桥接"到"合并"的认知跃迁

桥接思维：**life 是后端，DFC 是前端，两者通过 API 通信。**
- 问题：前端永远追不上后端的状态更新
- 问题：每次通信都要序列化/反序列化
- 问题：两套独立的上下文窗口 = 双倍 token

合并思维：**life 本身就是对话者，只是有两种对话对象——自己（心跳）和用户（聊天）。**
- 优势：零信息损失
- 优势：单一上下文窗口
- 优势：单次 LLM 调用

### 4.3 这不是"降级"DFC，而是"升级"对话能力

合并后对话能力对比：

| 能力 | DFC（当前） | LifeChatter（合并后） |
|------|-----------|---------------------|
| 人设一致性 | 依赖固定 prompt | SOUL.md + 连续内在状态 |
| 记忆准确性 | 需工具调用 | 原生访问记忆网络 |
| 情感连续性 | 无（每轮独立） | SNN + 调质层原生驱动 |
| 上下文深度 | 5-10 条消息 | 完整事件流 |
| 工具丰富度 | 5-8 个桥接工具 | 25+ 原生工具 |
| 响应延迟 | 2-3s（含 sub_agent） | 1-2s（省掉 sub_agent + 桥接） |
| Token 效率 | ~3600/轮 | ~1900/轮 |

## 五、风险与对策

### 5.1 上下文污染风险
**风险**：心跳的内在独白可能泄露到对话回复中  
**对策**：对话模式使用独立的 prompt 框架，明确指示"你正在和用户对话"而非"你在进行内在思考"。心跳独白作为背景上下文注入，但明确标注为"内在状态，不要直接输出"

### 5.2 心跳被对话阻塞
**风险**：对话模式执行时心跳暂停  
**对策**：心跳和对话异步并行。对话不进入心跳循环，而是通过独立的 Chatter 执行。两者共享状态但不共享执行线程

### 5.3 框架兼容性
**风险**：Neo-MoFox 框架要求 Chatter 组件在独立插件中  
**对策**：life_engine 插件同时注册 Service + Chatter 组件。框架已支持一个插件注册多种组件

### 5.4 渐进迁移
**风险**：一次性替换风险太大  
**对策**：Phase 0 先在 life_engine 中实现 LifeChatter，与 DFC 并存。通过配置切换。验证稳定后再移除 DFC

## 六、总结

**DFC 的问题不是实现不好，而是架构定位错误。** 当 life_engine 已经是一个拥有完整认知能力的自主系统时，再用一个传统聊天机器人做"前台接待"，就像让一个只会照本宣科的前台代替专家和客户沟通——无论你给前台多少小纸条，她的表达永远比不上专家自己开口说话。

**合并的本质是：让专家自己开口。**

Life 不需要"翻译"来和用户对话。她需要的是一个"对话模式"——切换提示词框架，打开 send_text 工具，然后用她自己的方式说话。
