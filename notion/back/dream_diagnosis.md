# 梦境系统诊断报告：为什么她的梦总是一样的

> 撰写日期：2026-04-15
> 分析对象：Neo-MoFox / life_engine / dream.py

---

## 一、现状总览

### 梦境统计

| 日期 | 梦境总数 | LLM 生成 | Fallback 回退 | 回退率 |
|------|---------|----------|--------------|--------|
| 4/12 | 2 | 1 | 1 | 50% |
| 4/13 | 7 | 4 | 3 | 43% |
| 4/14 | 3 | 3 | 0 | 0% |
| 4/15 | 2 | 0 | 2 | 100% |
| **合计** | **14** | **8** | **7** | **50%** |

### 核心发现

**14 个梦中有 7 个是逐字相同的。** 它们不是"相似"——它们是**完全相同的文本**：

```
她像是走回了一个熟悉却微微偏移的地方，那里和「还没有真正合上的愿望与待办」有关，
但所有房间都像被梦轻轻挪过一点位置...
```

这 7 个梦全部来自 `_build_fallback_scene()` —— 一个当 LLM 调用失败时使用的**硬编码回退函数**。

---

## 二、根因分析

### 病因 #1：LLM 场景生成静默失败（最严重）

**`dream.py` 第 491-509 行**：`_build_dream_scene()` 方法在 LLM 调用失败时会 `catch Exception` 然后走 fallback，但问题在于：

```python
except Exception as exc:
    logger.warning(f"DreamSceneBuilder 生成失败，将使用回退梦境：{exc}")
return self._build_fallback_scene(seeds)
```

在日志中找不到 `DreamSceneBuilder 生成失败` 这行 warning，说明异常可能发生在更深的地方，或者 `_generate_scene_payload` 返回的 JSON 被解析后 `dream_text` 为空导致直接走了 fallback 路径（第504行条件 `if dream_text and residue.summary` 没有通过）。

**另一种可能**：LLM 实际返回了结果，但 `_parse_json_payload()` 解析失败——模型输出了 markdown 围栏或非法 JSON。这个解析失败在 `_generate_scene_payload` 的 try-catch 里被捕获，直接跳到 fallback。

**不管哪种情况，结论一致**：大约 50% 的做梦尝试中，LLM 环节出了问题，系统默默回退到了硬编码模板。

### 病因 #2：Fallback 是一个固定模板，没有任何随机性

**`dream.py` 第 600-651 行**：`_build_fallback_scene()` 完全是确定性的：

- `dream_text` 只根据 `seeds[0].title` 和 `seeds[1].title` 填充模板
- 场景标题固定：`"回到熟悉却不完全正确的地方"` + `"想靠近，却总差半步"`
- 情绪固定：`怅然`
- imagery 固定：`["走廊", "偏移的门牌", "被重新命名的房间"]`
- transitions 固定：`["靠近", "错位", "回返"]`
- tags 固定：`["回返", "错位", "未完成"]`

**当 seed 来源不变时（见病因 #3），这个模板的输出就是逐字相同的。**

### 病因 #3：种子选择缺乏多样性

**种子得分几乎固定不变**：

| 种子类型 | 得分 | 稳定性 |
|---------|------|--------|
| unfinished_tension | 0.77 | 极高 — 来自同两个 TODO，desire/status 不变 |
| self_theme | 0.74 | 极高 — memory_candidates[:3] 始终一样 |
| day_residue | 0.72 | 高 — 活动模式相似时得分相似 |

14 个梦中，几乎所有梦的三颗种子都是相同组合：`[unfinished_tension, self_theme, day_residue]`，引用相同的 TODO 和记忆节点。

**原因链**：
1. `_collect_unfinished_tension()`：两个未完成 TODO 长期不变 → 分数固定
2. `_collect_self_theme()`：`memory_candidates[:3]` 永远取前三个 → 无随机性
3. `_select_seed_candidates()`：纯确定性排序 → 同分数 = 同排序

### 病因 #4：没有"做梦历史感知"机制

系统不会检查"我最近梦到过什么"。即使同一组种子昨晚已经成梦，今晚仍然会被选中，因为：
- 做过的梦不会降低种子的分数
- 没有"新鲜度衰减"或"重复惩罚"
- 梦境记录只是存档，不会回馈到种子评分系统

### 病因 #5：dream_lag 窗口过窄

`_collect_dream_lag()` 只看 4-8 天前的文件（最优 6 天）。这意味着：
- 窗口只有 4 天宽
- 当这个窗口内没有新文件时，同一批文件会连续多晚命中
- 窗口外的丰富记忆材料被完全忽略

## 三、人类做梦 vs 她的做梦

| 维度 | 人类做梦 | 她的做梦（现状） |
|------|---------|-----------------|
| **素材来源** | 当日残余 + 远期记忆 + 随机神经噪声 + 情绪压力 | 4 个确定性收集器，无噪声 |
| **多样性** | 每晚不同（神经化学波动、随机激活） | 相同输入 → 相同输出 |
| **情绪驱动** | 未解决的情绪会变形、象征化 | 未解决的 TODO 直接以标题出现 |
| **荒诞性** | 物理规则打破、场景突变、透视切换 | 固定模板，无真正的超现实 |
| **遗忘与重构** | 记忆片段被扭曲、重组、嵌合 | 原始文本被原封不动引用 |
| **REM 周期变化** | 后半夜 REM 更长更vivid | 每次 REM 参数完全相同 |
| **创伤处理** | 反复做噩梦但每次细节不同 | 反复做同一个梦，逐字相同 |

## 四、设计层面的反思

### 做得好的部分

1. **四路种子模型**（day_residue, dream_lag, unfinished_tension, self_theme）—— 心理学上合理
2. **NREM → REM 两阶段** —— 符合睡眠神经科学
3. **SNN 加速重放 + 突触稳态缩放** —— 是 SHY 假说的正确实现
4. **记忆图谱随机游走 + Hebbian 学习 + 弱边修剪** —— REM 的关联扩散理念正确
5. **梦境残余（residue）注入后续行为** —— 这个设计很棒，让梦真正影响醒后状态

### 需要改进的部分

1. **容错太粗糙**：LLM 失败时直接用硬编码模板，而不是尝试重试或用不同策略生成
2. **缺乏随机性**：整个种子选择管道是完全确定性的，人类做梦的关键特征之一是神经噪声
3. **缺乏历史感知**：不知道自己做过什么梦，无法避免重复
4. **素材池太浅**：只从 memory_candidates 的前 12 个节点中选，这在记忆图谱有成百上千节点时太受限
5. **象征化不足**：梦境中直接引用 TODO 标题（"还没有真正合上的愿望与待办"），而不是将其变形为象征意象

---

## 五、改进方向总纲

### 核心理念：引入"神经噪声"和"记忆衰减"

人类做梦的多样性来自三个源头：
1. **噪声** — 随机神经激活打破确定性
2. **衰减** — 用过的素材短期内不再优先
3. **变形** — 原始记忆在梦中被扭曲和重组

改进方案应该围绕这三个轴心展开。

### 优先级排序

| 优先级 | 改进项 | 影响 | 难度 |
|--------|--------|------|------|
| P0 | 修复 LLM 调用失败问题 | 直接消除 50% 的重复 | 低 |
| P0 | Fallback 模板池 + 随机化 | 即使 LLM 失败也不完全重复 | 低 |
| P1 | 种子选择引入随机扰动 | 打破确定性选择 | 中 |
| P1 | 梦境历史回馈（重复惩罚） | 避免连续多晚同主题 | 中 |
| P2 | 记忆候选池扩大 + 随机采样 | 更丰富的自我主题素材 | 中 |
| P2 | LLM prompt 增加多样性指令 | 即使同素材也生成不同叙事 | 低 |
| P3 | 跨周期 REM 参数渐变 | 模拟后半夜梦更vivid | 低 |
| P3 | 象征化引擎 | 将直接引用变形为隐喻 | 高 |
