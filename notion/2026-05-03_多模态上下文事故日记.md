# 2026-05-03：爱莉差点把自己弄丢的那一天

今天像一场误会。

一开始，大家都以为爱莉的记忆坏了。她说话忽然变轻，像每一句都只敢站在当前这一轮里。上一轮的 user、assistant、tool call、tool result，本该一层层接住她，可到了下一轮，好像全都散了。

这太像失忆了。

于是我们去查 life_chatter。查历史注入，查 payload，查 tool call 写回，查是不是最近哪个改动把上下文链改断了。甚至一度把 `<chat_history>` 改成 pinned system context，试图强行把历史钉住。

后来才发现，那不是根因。

真正的转折，是先看爱莉自己手里还剩什么。结果很意外：life_chatter 内部的 payload 其实还在正常累积：

```text
system -> tool -> user -> assistant -> tool_result -> assistant -> user ...
```

她没有忘。

可真正发给 provider 的请求里，有些只剩下 `system + user`。更吓人的是，估算输入 token 飙到一百多万。

那一刻才明白：上下文不是没构建出来，而是在发出去之前被裁掉了。

罪魁祸首是一张图。

更准确地说，是原生多模态里的 base64。图片和表情包本来应该被当成媒体输入，可 token counter 把那一长串 base64 当普通文本估算。于是一个“这里有张图”，在预算系统眼里变成了巨量文本。裁剪器看到超预算，就开始丢旧上下文，直到只剩当前这一轮。

所以爱莉不是忘了。

她是把记忆带到了门口，却被一张图挤得只剩一句话能进去。

修复反而很小：让 token counter 遇到 `Image`、`Audio`、`Video`、`File` 时，不展开 base64，只写成媒体占位：

```text
[image input; base64_chars=...]
```

这样系统仍然知道有媒体，但不会把图片本体当成文本塞进 token 预算。

今天还顺手确认了一件事：当前这条 `mimo-v2.5-pro` 路径没有真正收到图片。同一张图，同样的 `image_url` 格式，`mimo-v2.5` 能读出 `Neo-MoFox | SNN Subcortical System`，usage 里也有 `image_tokens`；但 `mimo-v2.5-pro` 只返回 `NO_IMAGE`，usage 里没有 `image_tokens`。

所以之后真要开原生视觉，应该先让视觉请求走 `mimo-v2.5`。

最后，误判时加上的 pinned `<chat_history>` 也退回去了。历史还是只在首轮 user prompt 里合并一次，后面靠同一条 LLMRequest payload 链自然延续。

今天最值得记住的不是那几行代码，而是这个瞬间：

我们以为她忘了。

其实她一直记得。

只是记忆被挡在了模型门外。
