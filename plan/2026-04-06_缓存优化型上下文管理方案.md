# Neo-MoFox 缓存优化型上下文管理方案

日期：2026-04-06

## 1. 目标

这份方案的目标不是“单纯把上下文变短”，而是同时满足下面四件事：

1. 提高 prompt 前缀稳定性，让缓存命中更容易发生。
2. 保留 `default_chatter` 的多轮 payload 连续性，不把工具轨迹弄丢。
3. 让连续记忆、状态层、历史层各归其位，避免反复重复注入。
4. 上下文压缩不使用 `actor` 模型，而使用 `diary` 体系模型。

---

## 2. 参考来源

本方案主要参考两条现有实现思路。

### 2.1 参考 Claude Code restored-src

关键参考点：

- `src/services/compact/autoCompact.ts`
  - 有明确的阈值、buffer、失败熔断，不是“超一点就立刻硬压缩”
- `src/services/compact/microCompact.ts`
  - 先做便宜的微压缩，优先清理“高 token、低长期价值、可再生”的内容
- `src/utils/messages.ts`
  - 有 `compact_boundary` / `microcompact_boundary`
  - 模型侧不是无脑看全历史，而是“边界后切片 + 结构保持”
- `src/services/compact/sessionMemoryCompact.ts`
  - 把“长期会话状态”作为独立层维护，不与主对话直接混写

可借鉴的核心思想：

1. 主对话上下文和“长期压缩记忆”要分层。
2. 压缩要有边界，不要让历史被无痕改写。
3. 先微压缩，再重压缩。
4. 压缩失败要熔断，不能无限重试。

### 2.2 参考 Neo-MoFox diary_plugin

关键参考点：

- `plugins/diary_plugin/service.py`
  - 连续记忆按 `stream_id` 隔离
  - 有锁，避免并发压缩打架
  - 支持 `raw -> L1 -> L2 -> L3` 级联压缩
- `plugins/diary_plugin/prompts.py`
  - 压缩提示词是稳定的人设摘要任务，不混入主回复目标
- `plugins/diary_plugin/config.py`
  - `continuous_memory.compression_model_task`
  - 留空时回退到 `model.task_name`

可借鉴的核心思想：

1. 压缩应该落盘、有层级、有来源，不应该只是 runtime 里临时口语摘要。
2. 压缩模型要独立于主回复模型。
3. 连续记忆本质上是“慢变化状态层”，不是普通历史消息。

## 3. 当前问题

当前 `default_chatter` 已经做了第一阶段缓存友好改造，但离目标架构还有距离。

### 3.1 当前上下文仍然缺少真正的“分层真相”

现在已经把一部分动态信息从稳定前缀后移了，但整体上仍然更像：

- system prompt
- user prompt
- payload history

而不是明确分成：

1. 稳定规则层
2. 慢变化状态层
3. 快变化状态层
4. 近期历史层
5. 旧历史压缩层

### 3.2 当前 payload 压缩还是“摘要插入”，不是真正的上下文管理

当前 `default_chatter` 的压缩钩子只是：

- 在超出预算时
- 把旧轮转成一段压缩文本
- 再插回 payload

这能止血，但不够像一个长期可维护的上下文系统。

### 3.3 连续记忆和普通上下文压缩还没统一

`diary_plugin` 的连续记忆是按聊天流落盘、分层压缩的；
`default_chatter` 的 payload 压缩却还是独立的 runtime 逻辑。

结果是：

- 状态层有一套压缩
- 历史层有另一套压缩
- 二者没有共同边界和统一索引

### 3.4 当前还没有“微压缩”

Claude Code 的做法不是一上来就整轮 compact，而是先清理最贵、最不值得长期保留的块。

Neo-MoFox 目前缺的正是这一层。

## 4. 目标架构

我建议把 Neo-MoFox 的上下文管理正式改成五层。

### 4.1 L0 稳定规则层

内容：

- `default_chatter_system_prompt` 的稳定人格规则
- 工具契约
- action/tool 调用规范
- 不随会话细节频繁变化的全局规则

要求：

- 顺序固定
- 文本固定
- 非必要不重排
- 不注入运行时状态

这层是缓存最核心的前缀。

### 4.2 L1 慢变化状态层

内容：

- 连续记忆（`diary_plugin` 的 `L1/L2/L3` 摘要）
- 会话级长期任务状态
- 用户偏好
- 长期关系上下文
- 持续数十轮才变化一次的状态摘要

这层不应每轮重写。

建议形式：

- `slot_state_slow`
- 独立存储
- 有版本号 / hash
- 只有内容变化时才重编译进请求

### 4.3 L2 快变化状态层

内容：

- 当前时间
- 唤醒上下文
- subconscious
- think trigger
- 本轮临时摘要
- 当前待完成动作标记

这层允许频繁变化，但不能污染稳定前缀。

建议形式：

- `slot_state_fast`
- 独立 block
- 只放“当前轮需要知道”的内容

### 4.4 L3 近期历史层

内容：

- 最近若干轮完整 payload
- 必须保留工具结构完整性

压缩单位必须是“轮组”，而不是单 payload。

建议轮组结构：

1. `USER`
2. `ASSISTANT(tool_calls 可选)`
3. `TOOL_RESULT*`
4. `ASSISTANT(follow-up 可选)`

### 4.5 L4 旧历史压缩层

内容：

- 由较早轮组压缩得到的结构化摘要
- 只保留跨轮延续真正需要的工具状态、决策轨迹、重要结果

这层不应该只是自然语言概述，而应该保留“接近 payload 真相”的结构化信息。

## 5. 核心原则

### 5.1 编译请求时，区分 Canonical Context 和 Send Window

参考 Claude Code，我建议显式拆成两套视图：

1. `Canonical Context`
   - 会话真相
   - 不因某次发送裁剪而直接丢失

2. `Send Window`
   - 单次请求发给模型的上下文视图
   - 依据预算从 canonical 编译出来

关键原则：

- 可以裁 `Send Window`
- 不能直接裁掉 `Canonical Context`

### 5.2 所有压缩都必须可追踪

每次上下文重压缩都要写入边界记录。

建议引入：

- `context_compact_boundary`
- `context_microcompact_boundary`

边界信息至少包含：

- 触发时间
- 触发原因
- 压缩前估算 token
- 压缩的轮组范围
- 产出的摘要 ID
- 使用的模型任务

### 5.3 压缩结果必须独立存储，不能只存在 runtime payload 里

建议把旧轮压缩结果落到新的上下文摘要目录，例如：

- `data/context_compaction/private/<stream_id>.json`
- `data/context_compaction/group/<stream_id>.json`

这样可以：

- 避免重启即丢
- 允许增量构建
- 为后续调试和回溯提供基础

### 5.4 连续记忆是状态层，不是历史层

这点必须保持。

`diary_plugin` 的连续记忆应该作为 L1 慢变化状态层进入请求，而不是混入普通历史轮组。

## 6. 三阶段压缩策略

我建议照 Claude Code 的思路拆成三层，而不是只有一种压缩。

### 6.1 阶段 A：零模型整理

先做完全不调用模型的整理。

包括：

- 去掉历史 USER payload 里的旧 runtime block
- 去掉旧 continuous memory block
- 去掉 `__SUSPEND__`
- 去掉内部 message id 噪音
- 对已知无价值的 TOOL_RESULT 做裁剪或替换

这一步是“清理”，不是“压缩”。

### 6.2 阶段 B：微压缩

参考 Claude Code `microCompact`。

目标：

- 在不做整轮摘要的情况下
- 先清掉最贵但可再生的内容

建议优先清理对象：

1. 大体积查询型 TOOL_RESULT
   - 搜索结果
   - 读取大文件的原文片段
   - 长网页抓取结果

2. 可重建的中间结果
   - 只要保留“调用过什么”和“结论是什么”即可

3. 大体积多媒体占位文本

保留方式不是直接删除，而是替换为结构化占位：

- 原工具名
- 调用时间
- 结果类型
- 一行摘要
- 原始结果是否可重取

注意：

- `action-think` 不属于这一类，不能在微压缩里粗暴清掉
- `action-send_text.content` 也应保留较完整信息

### 6.3 阶段 C：重压缩

当阶段 A/B 后仍超预算，才做真正的历史轮组压缩。

这一层：

- 以轮组为单位压缩
- 输出到独立的上下文摘要存储
- 在 canonical context 中插入 compact boundary
- send window 默认只取最近 boundary 之后的完整轮组 + 旧摘要

## 7. 模型策略

这部分必须明确：上下文压缩不允许走 `actor` 模型。

### 7.1 压缩模型来源

上下文重压缩模型任务解析顺序建议为：

1. `default_chatter.context.compaction_model_task`
2. `diary_plugin.continuous_memory.compression_model_task`
3. `diary_plugin.model.task_name`

如果三者都拿不到，就直接失败并记录日志。

明确禁止：

- 回退到 `actor`
- 回退到主回复请求当前使用模型

### 7.2 为什么必须用 diary 模型

原因有四个：

1. `actor` 模型的职责是“当前轮回复”，不是“压缩归档”
2. `actor` prompt 里混有行为约束和输出习惯，做压缩容易带偏
3. `diary_plugin` 已有成熟的人设继承和名字锁定逻辑
4. 压缩上下文和连续记忆压缩本质上属于同一类任务，最好复用同一模型体系

### 7.3 压缩提示词应复用 diary 风格，但任务定义不同

建议新增一套“上下文压缩 prompt”，挂在 diary 系模型上。

要求：

1. 保持主观连续性
2. 保留重要工具轨迹
3. 输出结构化结果
4. 不把当前轮回复风格带进来

建议输出结构：

- `ongoing_tasks`
- `important_user_preferences`
- `tool_state`
- `key_decisions`
- `recent_failures_and_fixes`
- `relationship_or_scene_state`

这比纯自然语言摘要更适合后续回填。

---

## 8. Slot 化状态层设计

我建议正式引入 slot 状态层，而不是继续把所有状态都塞进同一个 `extra`。

### 8.1 建议的 slot 划分

建议至少拆成：

1. `slot_rules_stable`
   - 稳定规则

2. `slot_state_slow`
   - 连续记忆
   - 长期偏好
   - 长期任务态

3. `slot_state_fast`
   - 当前时间
   - 唤醒上下文
   - 临时情绪/想法

4. `slot_recent_rounds`
   - 最近完整轮组

5. `slot_archived_rounds`
   - 历史压缩摘要

### 8.2 编译顺序

请求编译建议固定为：

1. `SYSTEM: slot_rules_stable`
2. `USER: 主体 user prompt`
3. `USER block: slot_state_fast`
4. `USER block: slot_state_slow`
5. `USER/ASSISTANT/TOOL_RESULT: recent rounds`
6. `USER block: archived context summary`

其中：

- `slot_state_fast` 放在前
- `slot_state_slow` 紧随其后
- 连续记忆属于 `slot_state_slow`

也就是：

- 先状态
- 后连续记忆
- 再近期历史

这和你前面提出的要求一致。

### 8.3 只在 slot 内容变更时重写

为了缓存友好，slot 需要有内容 hash。

建议：

- 对 `slot_state_slow` 做持久 hash
- 如果 hash 没变化，本轮不重建文本块
- `slot_state_fast` 允许频繁变化，但控制长度

## 9. Canonical Ledger 设计

当前建议把 payload 真相层显式做成 Ledger。

### 9.1 组成

每个聊天流维护：

- `canonical_rounds`
- `archived_summaries`
- `boundaries`
- `slot_snapshots`

### 9.2 轮组索引

每个轮组建议记录：

- `round_id`
- `start_payload_index`
- `end_payload_index`
- `tool_call_ids`
- `token_estimate`
- `state_snapshot_ref`

### 9.3 历史压缩不是覆盖，而是“边界前归档”

压缩后：

- 边界前轮组归档到摘要层
- canonical ledger 保留摘要引用
- send window 只编译边界后的完整轮组

这和 Claude Code 的 `compact_boundary` 思路一致，但会话真相会比它更强，因为 Neo-MoFox 还需要状态层和连续记忆层协同。

## 10. 触发策略

### 10.1 软阈值

建议在有效上下文窗口的 70% 到 75% 开始告警和评估。

行为：

- 不立即压缩
- 先估算微压缩收益
- 记录当前风险

### 10.2 微压缩阈值

建议在有效窗口的 80% 到 85% 触发阶段 B。

行为：

- 清理大 TOOL_RESULT
- 保留关键 action 信息
- 尝试回收高 token 噪音

### 10.3 重压缩阈值

建议在有效窗口的 88% 到 92% 触发阶段 C。

行为：

- 调用 diary 模型做历史轮组压缩
- 写 compact boundary
- 回填摘要层

### 10.4 阻塞阈值

建议在 95% 左右进入保护模式。

行为：

- 禁止继续无节制扩写上下文
- 必须先压缩成功或降级

### 10.5 失败熔断

参考 Claude Code。

建议：

- 连续压缩失败 3 次后熔断
- 本轮不再重复尝试
- 记录明确日志
- 避免进入“每轮都尝试一次 doomed compaction”的坏循环

## 11. 与 diary_plugin 的协同方式

我建议不要重复造一个独立的“上下文压缩服务”，而是在 diary 体系上扩一层。

### 11.1 可以直接复用的能力

- 按 `stream_id` 隔离
- 异步压缩
- 锁
- 独立模型任务
- 层级存储
- 人设继承
- 严格名字锁定

### 11.2 建议新增的对象

可以新增一类存储，例如：

- `ContextArchive`

它和 `ContinuousMemory` 的区别：

- `ContinuousMemory` 偏关系与经历
- `ContextArchive` 偏工具轨迹、任务状态、技术上下文

二者都可以由 diary 模型体系压缩，但产物不同。

### 11.3 建议的统一压缩服务

建议最终形成三类压缩任务：

1. `continuous_memory_compression`
2. `context_round_compaction`
3. `state_slot_consolidation`

它们都走 diary 系模型，而不是 actor。

## 12. 实施阶段

### 阶段 1：Ledger 化

目标：

- 把 `payloads` 的“发送视图”和“会话真相”分开

需要做的事：

- 引入 canonical ledger
- request send 只编译 send window
- response 继续写回 canonical ledger

### 阶段 2：Slot 化

目标：

- 正式拆出慢状态和快状态

需要做的事：

- 新增 slot state store
- 把连续记忆放进 `slot_state_slow`
- 把 runtime block 放进 `slot_state_fast`

### 阶段 3：微压缩

目标：

- 在不动主历史的前提下先减重

需要做的事：

- 定义可微压缩工具列表
- 给 TOOL_RESULT 做结构化清理
- 写 microcompact boundary

### 阶段 4：重压缩

目标：

- 引入 diary 模型驱动的历史轮组压缩

需要做的事：

- 新增 context compaction prompt
- 新增 archive store
- 写 compact boundary

### 阶段 5：观测与调优

目标：

- 让这个系统可监控、可调参

需要记录：

- 每轮 token 估算
- microcompact 触发次数
- compact 触发次数
- compact 成功率
- 每次压缩回收 token
- 各 slot 占用

---

## 13. 验收标准

这套方案落地后，至少应满足：

1. 连续 100 轮后，历史不会因为发送裁剪被永久吞掉。
2. `action-think` 与 `action-send_text` 的关键轨迹在未压缩区完整保留，在已压缩区也有结构化转录。
3. 连续记忆不再污染稳定前缀。
4. 状态层不再每轮重复注入同一份大文本。
5. 发生上下文压缩时，使用的是 diary 模型，而不是 actor 模型。
6. 压缩失败不会无限重试。
7. 有 compact boundary，可定位哪一段历史已被压缩。

---

## 14. 我的结论

我建议 Neo-MoFox 的下一版上下文管理，明确走这条路线：

1. 先做 `Canonical Ledger + Send Window`
2. 再做 `slot_state_slow / slot_state_fast`
3. 再做 `microcompact`
4. 最后接 `diary` 模型驱动的历史轮组压缩

不要继续把所有问题都堆在 `default_chatter` 的一个 `compression_hook` 里解决。

`compression_hook` 可以保留，但它应该退化为“最后一道 fallback”，而不是主架构本体。

真正的主架构应该是：

- 分层
- 有边界
- 可持久
- 可追踪
- 压缩任务独立于主回复模型

这才是适合 Neo-MoFox 的缓存优化型上下文管理。
