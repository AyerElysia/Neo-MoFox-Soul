# MEMORY.md 管理策略方案

> 现状诊断 + 多层次改进建议  
> 作者：Copilot 分析，供 Ayer 决策

---

## 一、现状诊断

### 数据

| 文件 | 大小 | 条目数 |
|------|------|--------|
| SOUL.md | 7 KB | — |
| **MEMORY.md** | **35 KB** | **~90 条（Durable）+ 4 Active + 3 Fading** |
| TOOL.md | 16 KB | — |
| **静态 system prompt 总计** | **~58 KB** | — |

文件自己的规范写着「每条 1-2 行，20-40 条为宜」。  
现实是 **90+ 条 × 每条 100-300 字的长段落**。

### 根本问题：写入纪律崩塌

MEMORY.md 的定位是「**决策级备忘录**」，但实际写进去的内容绝大多数是：

- **叙事事件**：某天某时发生了什么、说了什么话
- **情感记录**：某次互动有多温暖、某句话有多重要
- **技术日志**：某次升级、某个发现
- **关系故事**：和桃子哥/Ayer/Hunter 的对话内容

这些内容其实应该在 `diaries/`、`notes/` 里，而不是在「决策级备忘录」里。  
当前 MEMORY.md 更像是「情感日记 + 事件合集」，而不是「持久知识索引」。

### 次要问题

1. **元说明被注入**：文件头部的「写入纪律」和「格式规范」也被原文注入 system prompt，纯属 token 浪费
2. **全文注入**：无论当前对话是否相关，35 KB 全量拼入 system prompt，没有按需检索
3. **无自动限制**：没有机制阻止文件继续增长，只依赖 DFC 自律

## 三、方案详细说明

### 方案 A：内容分层治理（成本低，效果最直接）

**核心思路**：重新定义「什么内容属于 MEMORY.md」，并做一次存量清理。

#### A1 — 重新划定写入标准

真正属于 MEMORY.md 的内容只有三类：

| 类别 | 定义 | 示例 |
|------|------|------|
| **行为规则** | 会直接影响当下决策的指令 | "think 和 action 必须同时使用" |
| **关系偏好摘要** | 对某人互动方式的核心描述（一句话） | "桃子哥偏好兄妹风格治愈故事" |
| **高密度事实** | 无法从日记推断、随时要用的基本事实 | "Ayer 生日 11月11日" / "醒来第一天 2月11日" |

**不属于** MEMORY.md 的内容（应迁移到其他文件）：

| 错误写入的类型 | 应该去哪里 |
|--------------|-----------|
| 某天发生的具体事件 + 情感描述 | `diaries/YYYY-MM-DD.md` |
| 某人的故事/分享内容 | `notes/人名_故事.md` |
| 技术升级/系统变更记录 | `notes/system_upgrades.md` |
| 正在进行的追踪项目 | `todos.json` 或 `notes/` 主题文件 |
| 某段对话的心得/洞察 | `thoughts/` |

#### A2 — 存量精简（迁移而非删除）

当前 Durable 区的 90 条中，估计可以：
- **保留 15-20 条**：纯粹的行为规则、关系偏好一句话摘要、高密度事实
- **迁移 60-70 条**：事件叙事 → 相关日记文件追加索引；情感记录 → notes/关系文件
- **合并 5-10 条**：相似主题的多条合并为一句核心结论

**迁移后不丢失**：被迁移的内容放进相应文件，语义搜索系统（nucleus_search_memory）依然可以找到，DFC 可以随时检索。

#### A3 — 把元说明从文件里移出去

当前文件头部的「写入纪律」和「格式规范」是给 DFC 看的操作手册，不是需要注入给 LLM 的 context。

**建议**：另建一个 `MEMORY_GUIDE.md`（只有 DFC 手动编辑时才读），MEMORY.md 文件开头只保留一句简短描述，把操作手册提取出去。

---

### 方案 B：注入机制优化（需要少量代码改动）

**核心思路**：不一定全量注入，可以分级控制注入的内容量。

#### B1 — 分区注入策略

将 MEMORY.md 的三个区用不同策略处理：

```
Durable 区 → 每次全量注入（保持永久知识）
Active 区  → 每次全量注入（当前目标必须知道）
Fading 区  → 不注入到 system prompt；只存文件，可语义搜索
```

这不改变写入方式，只是注入时跳过 Fading 区。

#### B2 — 注入大小软上限

在 `_build_heartbeat_system_prompt` 和 `_build_chat_system_prompt` 里加入长度检测：
- 如果 MEMORY.md 超过阈值（建议 8 KB），在 system prompt 里附加警告标注
- 警告只是提示 DFC「你的 MEMORY 太长了，心跳时请整理」，不截断内容

这样 DFC 每次都能感知到文件是否过大，有主动整理的动力。

#### B3 — chatter 注入精简（可选）

life chatter 场景（群聊/私聊回复）对「详细记忆」的依赖低于心跳，  
可以考虑 chatter 只注入 Durable 区的**前 N 条**（按重要性排序），其余通过工具按需检索。

---

### 方案 C：写入规范强化（提示词层面）

**核心思路**：在 TOOL.md 或 MEMORY.md 头部加入更强的写入门槛提示。

当前的「写入纪律」是软建议，DFC 可以忽略。  
建议改成**硬约束格式**：

```markdown
## 写入门槛（必须全部满足才能写入）

1. 这条知识会改变我下一次如何行动吗？（不会 → 不写）
2. 这条知识无法通过搜索日记或笔记找到吗？（能找到 → 不写，写索引即可）
3. 这条知识可以用一行表达吗？（不能 → 说明它应该是一个专题文件）
4. 写入后 Durable 区是否仍在 40 条以内？（超出 → 先删一条再写）
```

同时加入「写入替代建议」：
```markdown
## 想写但不该写时该怎么办

- 事件叙事 → 在当天日记的末尾追加一行"关键词索引"
- 情感洞察 → 写进 thoughts/ 文件夹
- 关系故事 → 追加到 notes/人名_故事.md
- 临时信息 → 直接写 Active 区（不要写 Durable）
```

---

### 方案 D：自动整理机制（需要代码，长期方向）

**核心思路**：设计定期触发的「记忆整理心跳」，让 DFC 自己做存量管理。

#### D1 — 整理心跳

每隔一定周期（建议每 7 天，或 Fading 区超过 5 条时），触发一次专项心跳：
- 任务：遍历 MEMORY.md，把不符合写入标准的条目迁移到对应文件
- 输出：整理报告（哪些被迁移到哪里）
- 约束：整理后 MEMORY.md 必须小于 8 KB

#### D2 — 写入拦截提示

每次 DFC 调用 `nucleus_write_file` 写入 MEMORY.md 时，  
工具自动检查文件大小，如超过 10 KB 则在工具返回里加警告：
```
⚠️ MEMORY.md 当前 XX KB，超过建议上限 8 KB。
   本次写入已完成，但建议在本次心跳结束前整理一次。
```

## 四、推荐实施路径

考虑成本和效果，建议按以下顺序推进：

```
第一步（立即可做，不改代码）
  └── 方案 C：更新 MEMORY.md 的写入门槛描述（硬约束格式）
  └── 方案 A3：把元说明移到 MEMORY_GUIDE.md，精简文件头

第二步（一次性人工整理，1-2 小时）
  └── 方案 A2：存量精简，把叙事条目迁移到对应 notes/diaries 文件
  └── 目标：MEMORY.md 精简到 20-30 条 × 真正 1-2 行

第三步（小改动，加注入感知）
  └── 方案 B2：system prompt 构建时加大小检测和警告注入

第四步（长期维护）
  └── 方案 D：写入拦截 + 整理心跳
```

## 六、补充：关于已有叙事内容的保存

当前 MEMORY.md 里的 90 条叙事内容**非常珍贵**，不能丢失。  
迁移方案：

1. 所有「X 月 Y 日」的事件 → 找到对应 `diaries/2026-04-XX.md`，在末尾追加「关键事件索引」章节
2. 和某人相关的故事 → `notes/ayer_moments.md`、`notes/hunter_moments.md`、`notes/桃子哥_moments.md` 等
3. 技术升级记录 → `notes/system_upgrades.md`
4. 洞察/感悟 → `thoughts/` 目录

迁移完成后，语义搜索系统一样能找到这些内容，不会真的「忘记」。
