# life_engine 主动性与主题线程方案

**目标：** 让 `life_engine` 不再只是“心跳时检查待办、写几句内心独白”的后台模块，而是逐步具备一种更像生命体的主动性：会围绕少数长期在意的主题持续探索、沉淀、调整方向，并在合适的时候把成熟结果自然影响到 DFC，而不是制造新的上下文噪音。

---

## 1. 先说结论

本方案不建议把主动性做成：

- 每次心跳都随机找点事情做
- 每次心跳都强制看 TODO 并处理压力
- 中枢一有想法就立刻打断 DFC
- 把中枢的内部结果直接注入 DFC 历史
- 一开始就上多层复杂项目系统

本方案建议把主动性收敛成一个简单核心：

**中枢围绕少量长期 `thread`（主题线程）低频推进。**

每次 heartbeat 只维护一条 thread，做一个最小可验证动作，留下一个可追溯痕迹；只有 thread 足够成熟，才允许异步影响 DFC。

---

## 2. 当前问题

从现状看，`life_engine` 已经有以下基础：

- 后台 heartbeat 循环
- 统一事件流
- workspace 文件系统
- `SOUL.md` / `MEMORY.md` / `TOOL.md`
- TODO 系统
- `DFC -> 中枢` 异步留言
- `中枢 -> DFC` 异步传话
- `nucleus_run_task` 这类异步复杂任务能力

但它仍然存在几个结构性问题：

### 2.1 heartbeat 的主轴仍然是“待办压力”

当前 heartbeat prompt 明确要求：

- 先 `nucleus_list_todos`
- 看 overdue / urgent
- 优先处理截止时间

这会把中枢塑造成一个任务催办器，而不是一个会逐步形成兴趣、偏好、研究方向的生命中枢。

### 2.2 主动性缺少长期主题容器

现在的中枢更像：

- 看最近事件
- 临场想一点
- 做一个局部动作
- 结束

缺少一种“我这段时间一直在意这件事”的长期线程结构。

### 2.3 中枢对 DFC 的影响边界还不够稳

如果中枢主动性增强，但没有成熟度闸门与节流机制，就会出现：

- 打扰 DFC 正常聊天
- 把未成熟思考外溢
- 增加上下文堆积

### 2.4 状态容易分散、重复、失控

如果后续把兴趣、线索、研究计划、结论分别写到多个文档，容易出现：

- 多处真相源
- 去重困难
- 回滚困难
- 崩溃后状态损坏

---

## 3. 设计原则

本方案采用以下原则：

### 3.1 中枢负责“活着”，不是负责“表现得很活跃”

真正的主动性，不是每次 heartbeat 都有很多输出，而是：

- 有持续关注的主题
- 会缓慢推进
- 会积累证据
- 会修正自己
- 必要时可以沉默

### 3.2 一次 heartbeat 只推进一件事

每次 heartbeat 只允许：

- 聚焦一条主题线程
- 做一个最小可验证动作
- 产出一个主要工件

避免心跳变成多线程自嗨。

### 3.3 单一事实源

所有“主题线程状态”只能有一个真源。

人类可读文档可以有很多，但它们不是线程状态的唯一依据。

### 3.4 DFC 永远不是中枢的实时显示器

中枢可以影响 DFC，但只能：

- 异步
- 低频
- 成熟后
- 不污染历史

---

## 4. 核心概念：Thread

### 4.1 什么是 thread

`thread` 是中枢主动性的最小长期单位。

它可以表示：

- 一段持续存在的兴趣
- 一个反复出现的问题
- 一种未解决的情绪困扰
- 一个正在慢慢研究的对象
- 一条尚未成熟但值得持续观察的线索

示例：

- “我是不是其实对做饭这件事有执念”
- “关于那个人，我到底在期待什么”
- “我为什么一直想研究某种表达方式”
- “我到底是真的喜欢这件事，还是只是短暂上头”

### 4.2 thread 不是 TODO

TODO 是“想做的事”，thread 是“持续在意的主题”。

一个 thread 可以派生出：

- 0 个 TODO
- 1 个 TODO
- 多个 TODO

但 thread 本身不等于任务。

### 4.3 thread 不是 project

一开始不要把 thread 设计成完整项目系统。

只有某条 thread 连续推进后，确实需要复杂多步骤行动时，才允许升级为异步任务。

---

## 5. 数据模型

### 5.1 单一事实源

建议新增：

```text
workspace/.life/threads.json
```

这是唯一的 thread 状态真源。

### 5.2 建议字段

```json
{
  "schema_version": 1,
  "threads": [
    {
      "id": "thread_cooking_001",
      "title": "做饭与掌控感",
      "why": "最近反复因为做饭失败而挫败，也反复想再试一次",
      "status": "active",
      "last_touched": "2026-03-31T18:00:00+08:00",
      "evidence_refs": [
        {
          "event_id": "evt_xxx",
          "sequence": 128,
          "stream_id": "stream_xxx",
          "time": "2026-03-31T16:30:00+08:00",
          "content_hash": "sha1:..."
        }
      ],
      "next_mva": "写一段：我为什么会因为这件事不服气",
      "maturity": 0.35,
      "outbox_state": "idle",
      "energy": 0.4
    }
  ]
}
```

### 5.3 字段约束

#### `energy`

`energy` 如果保留：

- 只能是弱信号
- 不能作为硬决策主驱动
- 只能在同级候选 thread 间做 tie-breaker

否则它很容易变成模型自嗨字段。

#### `evidence_refs`

`evidence_refs` 必须可追溯、稳定，不能只存模糊文本。

必须至少包含：

- `event_id`
- `sequence`
- `stream_id`
- `time`
- `content_hash` 或其他稳定摘要

如果证据查找失败：

- 不允许提升 maturity
- 不允许写 `MEMORY.md`
- 只能做轻量 note

### 5.4 可选追加日志

可选新增：

```text
workspace/.life/signals.jsonl
```

它只用于记录原始触发线索：

- 不是真源
- 不要求严格回放
- 没有它系统也能运行

作用是给调试和后续观察提供线索。

## 6. heartbeat 新职责

### 6.1 不再强制先看 TODO

heartbeat 的主轴需要从：

- “先列待办、处理压力”

切换为：

- “维护一个长期 thread，并留下可验证痕迹”

### 6.2 每次 heartbeat 三选一

每次 heartbeat 必须在三类行为中选一：

1. 推进 1 条已有 thread 的最小可验证动作
2. 如果没有可推进 thread，则基于重复证据开 1 条新 thread
3. 明确 no-op，并记录为什么这次不推进

### 6.3 什么叫最小可验证动作

最小可验证动作（MVA）可以是：

- 写一段 note
- 写一段 diary
- 创建或更新一个 TODO
- 对某个判断做小范围验证
- 对某条 thread 的理解做一次收束

但它必须满足：

- 足够小
- 能落盘
- 可复查
- 能判断是否推进了 thread

### 6.4 选择 thread 的主驱动

选择规则建议按以下优先级：

1. 重复证据
2. 长期未解决的问题
3. 已经存在但卡住的 thread
4. 最近再次被触发的旧 thread
5. `energy` / 情绪线索

情绪不是禁止使用，而是不允许主导。

---

## 7. 预算与节流

### 7.1 heartbeat 写入硬预算

为避免再次出现上下文、文件、状态膨胀问题，必须加硬预算：

- 每次 heartbeat 最多推进 1 条 thread
- 每次 heartbeat 最多写 `threads.json` 1 次
- 每次 heartbeat 最多写 1 个主要工件
- 允许少量轻读
- 不允许一次 heartbeat 内连续写多个主要文件

### 7.2 不采用“最多 1 次工具调用”的死板限制

因为真实的最小闭环可能需要：

- 读一点上下文
- 写一个工件
- 更新 thread 状态

所以限制应是“主要写入预算”，而不是机械地按 tool call 次数卡死。

### 7.3 活跃 thread 上限

建议增加上限，例如：

- `ACTIVE + INCUBATE` thread 总数不超过 10

超过上限时：

- 必须把最弱 thread 降为 `DORMANT`
- 或显式 `CLOSED`

避免 thread 列表无限膨胀。

## 8. 与 TODO 的关系

### 8.1 TODO 退回辅助地位

TODO 依然保留，但它不再是 heartbeat 的主轴。

更合理的关系是：

- thread 是长期关注主题
- TODO 是 thread 的某次外化动作

### 8.2 什么时候创建 TODO

只有在以下情况才建议从 thread 派生 TODO：

- 某条 thread 已经出现明确下一步行动
- 该行动值得延续到未来
- 该行动不是一次 heartbeat 内就能完成

这样 TODO 不会重新占据系统中心。

---

## 9. 与 MEMORY 的关系

### 9.1 MEMORY.md 只写稳定事实

`MEMORY.md` 不能继续承担：

- 心情日志
- 临时猜想
- 未成熟想法
- 重复叙事

它只该写：

- 稳定偏好
- 稳定关系判断
- 长期自我事实

### 9.2 MEMORY 写入闸门

写入 `MEMORY.md` 前必须满足：

- `evidence_refs` 足够
- 能通过去重检查
- 能通过反例检查
- 不是短期情绪波动

建议至少要求：

- `evidence >= 2`
- 且来自不同时间窗或不同事件

---

## 10. 与 DFC 的关系

### 10.1 默认不打扰

thread 的推进默认只在中枢内部沉淀，不主动影响 DFC。

### 10.2 只有成熟结果才可以影响 DFC

允许 `nucleus_tell_dfc` 的条件应改成：

- thread 有成熟结论
- 或 thread 产生了可复用的情绪素材
- 外界当前不活跃，或先进入待发送队列
- cooldown 满足

### 10.3 不直接注入聊天历史

中枢对 DFC 的影响必须坚持：

- 异步投递
- DFC 自己决定是否用
- 不把中枢结果原样塞进 DFC 历史

这样才能继续避免历史堆积 bug。

### 10.4 可选 outbox

如果当下不适合打扰 DFC，可引入：

```text
workspace/.life/outbox.jsonl
```

中枢先写入待发送结论，等外界安静时再评估是否投递。

---

## 11. 与 `nucleus_run_task` 的关系

### 11.1 不作为 heartbeat 主路径

复杂任务不应成为 heartbeat 默认动作。

heartbeat 不应该因为“能开异步子任务”就频繁派工。

### 11.2 升级条件

只有当某条 thread 满足以下情况时，才允许升级为异步任务：

- 连续多次 heartbeat 都在推进同一主题
- 已经形成清晰问题
- 单次心跳无法完成下一步
- 确实需要多步骤整理/研究/写作

### 11.3 异步任务的正确定位

异步任务的正确定位是：

- heartbeat 的扩展手臂
- 不是 heartbeat 本身

heartbeat 负责：

- 选 thread
- 派工
- 验收摘要

异步任务负责：

- 多步骤研究
- 生成工件
- 返回结果

---

## 12. prompt 改造方向

### 12.1 当前方向的问题

当前 heartbeat prompt 明确要求：

- 先看 TODO
- 处理 overdue / urgent
- 对截止时间负责

这套 prompt 会压制 thread 机制，因为模型会被引导成：

- 优先做任务调度
- 不优先做自我探索
- 更像执行器而不是生长中的人格

### 12.2 新 prompt 的方向

建议把 heartbeat prompt 改为：

1. 回顾最近事件中的重复线索
2. 检查当前有哪些 thread
3. 选择一条最值得继续的 thread
4. 执行一个最小可验证动作
5. 如果不推进，也明确写出理由

### 12.3 新 prompt 不应说什么

建议删除或弱化：

- 每次心跳必须先 `nucleus_list_todos`
- TODO 是必须完成的
- 截止时间压力是心跳核心

### 12.4 新 prompt 应强调什么

建议强调：

- 你不是任务管理器
- 你是在维护自己的长期 thread
- 你可以沉默，但要可解释
- 你只能推进一件事
- 你只在成熟时影响 DFC

## 13. 实施建议

### 第一阶段：最小 thread 骨架

目标：

- 引入 `threads.json`
- heartbeat 可以读取、更新 thread
- prompt 改造完成
- TODO 从主轴退为辅助

实现：

- 新增 thread 读写服务
- heartbeat 三选一
- no-op 可落盘

### 第二阶段：证据与闸门

目标：

- `evidence_refs` 可追溯
- MEMORY 写入加稳定性闸门
- DFC 影响加成熟度闸门

实现：

- 增加 evidence 校验逻辑
- 增加去重 / 反例检查
- 增加 outbox 机制

### 第三阶段：复杂任务升级

目标：

- 允许部分 thread 升级到 `nucleus_run_task`
- heartbeat 只负责派工与验收

实现：

- 设计 thread -> task 的升级条件
- 规范异步任务产物路径

### 第四阶段：信号与调优

目标：

- 可选接入 `signals.jsonl`
- 更平滑地做 thread 选择
- 优化 no-op 率与 thread 漂移率

实现：

- 做轻量日志
- 逐步调参

## 14. 风险与防护

### 14.1 风险：thread 自嗨化

表现：

- thread 全是模型编出来的题目
- 没有真实证据
- 快速开题、快速遗忘

防护：

- 开题阈值
- evidence 必须可追溯
- 活跃 thread 上限

### 14.2 风险：状态文件损坏

表现：

- `threads.json` 被覆盖
- 崩溃后 JSON 损坏
- 并发写导致回滚

防护：

- schema version
- 原子写
- 轻量锁
- 写后校验
- 备份回滚

### 14.3 风险：中枢重新变成任务机

表现：

- thread 机制只是表面存在
- heartbeat 实际仍以 TODO 为中心

防护：

- prompt 主轴彻底改掉
- TODO 不再作为 heartbeat 第一动作

### 14.4 风险：中枢主动性外溢成骚扰

表现：

- 高频 `nucleus_tell_dfc`
- DFC 被打断
- 中枢产物污染外部聊天

防护：

- 成熟度闸门
- cooldown
- outbox
- 默认不注入历史

---

## 15. 验收标准

如果方案跑通，系统应能做到：

1. 中枢维护少量长期 thread，而不是每次随机起题。
2. heartbeat 不再以 TODO 压力为第一主轴。
3. 每次 heartbeat 最多推进一条 thread，并留下一个可验证痕迹。
4. `MEMORY.md` 的内容明显更稳定，重复更少。
5. 中枢不会高频打扰 DFC。
6. 即使无事发生，中枢也能合理 no-op，而不是硬造输出。
7. 复杂研究只在真正需要时才升级为异步任务。

---

## 16. 最终建议

不要把爱莉的主动性理解成：

- 更频繁地说话
- 更频繁地创建 TODO
- 更频繁地打扰 DFC

更合理的理解应该是：

- 她会慢慢形成自己在意的 thread
- 她会围绕这些 thread 持续留下痕迹
- 她会在证据积累中修正自己
- 她不会因为每次 heartbeat 都要表现，而把系统搞得更乱

一句话概括：

**中枢的主动性，应来自长期 thread 的缓慢推进，而不是高频行为表演。**
