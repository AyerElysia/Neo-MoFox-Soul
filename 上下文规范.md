# Neo-MoFox 上下文规范（唯一实现口径）

本文是 `default_chatter` 主链路的上下文注入与裁剪规范。  
目标：防止重复注入、避免上下文病态增长、提升缓存友好性。

## 1. 适用范围

- 主体插件：`plugins/default_chatter`
- 上下文内核：`src/kernel/llm/context.py`
- 连续记忆注入：`plugins/diary_plugin/event_handler.py`
- 潜意识唤醒消息来源：`plugins/life_engine/tools.py`

## 2. 核心原则（强约束）

- 稳定内容必须前置且稳定：`SYSTEM` / `TOOL` 前缀不可在轮次间无意义变化。
- 动态内容只放在当前轮用户输入附近，不得反复复制历史大块。
- 临时提示必须“注入后清理”，禁止长期残留在 payloads。
- 同一语义的消息不得重复注入（尤其是 life 唤醒、场景引导、连续记忆）。
- `system_reminder` 禁止注入到 `USER`；只允许作为前置 `SYSTEM` 块存在。

## 3. 请求结构规范

每个请求 payload 结构遵循：

1. `SYSTEM`（系统提示词、system_reminder、临时系统提示）
2. `TOOL`（工具 schema）
3. 对话链（`USER -> ASSISTANT(tool_calls) -> TOOL_RESULT -> ASSISTANT ...`）

结构校验由 `LLMContextManager._validate_payloads` 执行，必须满足：

- `tool_result` 必须紧跟 tool 调用链，不得直接接 `user`。
- 工具调用的 `call_id` 必须完整闭合。

## 4. system_reminder 规范

实现位置：`src/kernel/llm/context.py::_apply_reminders`

- reminder 由 `create_request(..., with_reminder=...)` 注册。
- 注入时先全量剥离旧 `<system_reminder>`，再插入新的前置 `SYSTEM` 块。
- 插入位置是最前面的 `SYSTEM` 区之后。
- 兼容清理历史遗留（旧实现曾把 reminder 塞进 `USER`）。

禁止事项：

- 禁止把 reminder 作为 `USER` 内容追加。
- 禁止多份 reminder 在 payload 中并存。

## 5. Enhanced 模式注入规范

实现位置：`plugins/default_chatter/runners.py::run_enhanced`

### 5.1 首轮与后续轮

- 首轮（`history_merged=False`）：
  - 注入 `history_text`
  - 注入 `extra`（仅行为约束增强与一次性补充）
- 后续轮（`history_merged=True`）：
  - `history_text=""`
  - `extra=""`（避免重复注入）

### 5.2 场景引导固定块

实现位置：`plugins/default_chatter/runners.py::_upsert_scene_guide_system_block`

- 场景引导不再放在 user extra。
- 场景引导作为独立 `SYSTEM` 固定块注入（`<session_scene_guide>`）。
- 该块处于前置 SYSTEM 区，避免每轮 user payload 重复携带。

### 5.3 未读消息写入方式

- 使用 `_upsert_pending_unread_payload`：
  - 若尾部已是 `USER`，直接 extend 到最后一个 `USER` payload。
  - 否则新建一个 `USER` payload。

### 5.4 临时 Life State

- 发送前临时注入 `<life_state>` 到当前尾部 `USER`。
- 发送完成后在 `finally` 中强制剥离（`_strip_temp_life_state`）。
- 该内容不得持久留在历史 payload 中。

### 5.5 think-only 临时系统提醒

- 当本轮仅调用 `action-think` 时，注入 `<temp_system_note>` 包裹的 `SYSTEM` 提示。
- 发送完成后在 `finally` 中清理（`_strip_temp_system_notes`）。
- 该内容是一次性控制提示，不允许跨轮累积。

## 6. 连续记忆规范

### 6.1 default_chatter 注入入口

实现位置：

- 固定块注入：`plugins/default_chatter/runners.py::_upsert_continuous_memory_system_block`
- 刷新逻辑：`plugins/default_chatter/runners.py::_refresh_continuous_memory_system_block`

规则：

- 连续记忆不再与 `history/unreads` 混在同一个 USER 文本内。
- 连续记忆作为独立 `SYSTEM` 固定块（`<continuous_memory_block>`）存在。
- 对遗留在 USER 内的旧连续记忆块会做迁移清理。
- 发送前会重排一次连续记忆块，确保它是“最后一个 SYSTEM”。

### 6.2 防重复策略

实现位置：`plugins/default_chatter/runners.py`

- 连续记忆按 `updated_at` 刷新 SYSTEM 固定块，不进行多份堆积。
- 连续记忆更新触发裁剪时，只裁最旧对话 payload，不回滚 `history_merged`，避免把整段历史重新灌入。

## 7. Life 唤醒消息规范

来源：`plugins/life_engine/tools.py` 会向 unread 队列写入 `life_nucleus_wake_*` 消息。

消费侧去重：`plugins/default_chatter/runners.py`

- 对 `is_life_engine_wake=True` 或 `sender_id=life_engine_nucleus` 的消息构建签名去重。
- 同签名重复消息立即 `flush_unreads` 清理，不再进入“新收到的消息”反复注入。
- 去重集合有上限（512）防止内存无限增长。

## 8. 梦境注入规范

实现位置：`plugins/life_engine/service.py`

- 做梦完成后（睡眠窗口 / 白天小憩 / 手动触发），会把完整 `DreamReport` 序列化为 `<dream_record>`。
- 梦境记录通过 `default_chatter.push_runtime_assistant_injection(...)` 进入 DFC 的 runtime assistant 队列。
- default_chatter 在 WAIT 阶段消费该队列并写入 `ASSISTANT` payload，自然参与后续对话推理。
- 梦境记录必须保留完整信息，不做摘要截断。
- 同一 `dream_id` 只注入一次（去重）。
- `get_state_digest_for_dfc()` 不再重复注入梦后余韵，避免和完整梦记录双重堆积。

## 9. 历史消息格式化规范

实现位置：`plugins/default_chatter/plugin.py::format_message_line`

- 以下 `message_id` 前缀视为内部噪声，不写入 prompt 历史：
  - `action_`
  - `api_`
  - `inner_monologue_`
  - `tool_`
  - `fc_tooluse_`

目的：减少无意义 token 与缓存抖动。

## 10. unread / history 语义

实现位置：`src/core/components/base/chatter.py`

- `fetch_unreads`：只读快照，不修改上下文。
- `flush_unreads(unread_snapshot)`：仅搬运传入快照中的消息到 history，不误清理新增未读。

这保证了并发场景下不会“读一批、清一堆”。

## 11. 缓存友好要求（工程约束）

- 稳定前缀（`SYSTEM` + `TOOL`）尽量不变。
- 场景引导、历史大块、连续记忆不得每轮重复注入。
- 临时块（`<life_state>`、`<temp_system_note>`）必须可追踪且可清理。
- 任何新注入器上线前，必须回答两个问题：
  - 是否会在同一请求生命周期内重复追加？
  - 是否会改变稳定前缀导致缓存失配？

## 12. 变更准入清单（必须满足）

修改上下文相关逻辑前后，至少验证：

1. 连续 3 轮对话中，`# 额外信息` 不应每轮重复出现（enhanced）。
2. 同一轮请求中，`<continuous_memory_block>` 作为独立 SYSTEM 固定块，且至多 1 个。
3. payload 中不存在多个 `<system_reminder>` 块。
4. `tool_result -> user` 结构错误不得出现。
5. life 唤醒重复消息不会反复出现在 `# 新收到的消息`。
