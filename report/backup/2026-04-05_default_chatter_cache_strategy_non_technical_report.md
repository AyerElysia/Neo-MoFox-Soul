# Default Chatter 缓存友好策略说明报告

日期：2026-04-05

## 1. 这份报告的定位

这是一份不讨论代码实现细节的说明报告，目标是把这次改造背后的思路讲清楚。

它回答的是这些问题：

- 为什么要做“缓存友好”改造
- 现在的问题到底出在哪里
- 这次调整的总体策略是什么
- 这样做会带来什么收益
- 还有哪些地方没有完全做完

## 2. 为什么要做这次改造

目前这套对话系统的问题，不是“模型不够聪明”，而是“每一轮给模型看的内容变化太大”。

如果一套提示词结构每次都在前面发生明显变化，会带来几个直接后果：

- 缓存命中率下降，成本和延迟都会上升
- 模型更难形成稳定的行为习惯
- 调试时不容易判断问题到底来自规则变化，还是来自用户输入变化
- 长对话会越来越重，最后要么变慢，要么被迫丢上下文

所以这次改造的核心目标，不是单纯减少字数，而是让“该稳定的东西稳定下来”，让“该变化的东西去后面变化”。

## 3. 这次发现的根本问题

这次分析后，发现问题主要集中在两个层面。

### 3.1 前缀不稳定

原本有很多会变化的提示信息，会在很靠前的位置进入模型输入。

这会导致模型每轮看到的“开头部分”都不一样。对于缓存和稳定性来说，这是最伤的一种情况。

可以把它理解成：

- 本来应该先让模型看到一套固定规则
- 结果每轮刚开头就被临时状态、时间、情绪、唤醒上下文之类的内容打断了

这样一来，系统虽然“信息很多”，但不够整洁，也不够稳定。

### 3.2 长对话缺少可持续的收纳方式

另一个问题是上下文会不断变长。

如果只是不断往后堆，早晚会遇到极限。到那个时候，如果系统只是简单丢掉更早的对话，就会出现两个问题：

- 模型突然忘掉一些重要背景
- 某些行为会显得前后不连贯

这说明问题不只是“超限”，而是“超限时怎么优雅收纳旧内容”。

## 4. 这次采用的总体思路

本次改造采用的是一种“分层整理”的思路。

简单说，就是把给模型看的内容分成三类：

### 4.1 稳定规则层

这类内容尽量保持不变。

它包括：

- 人设核心
- 表达风格
- 主要行为规范
- 工具使用契约
- 思考动作的稳定要求

这部分的目标是成为模型眼里的“固定骨架”。

### 4.2 运行时状态层

这类内容是会变化的，但又确实有价值。

它包括：

- 当前时间
- 临时提醒
- 潜意识状态
- 唤醒上下文

这类内容不应该放到输入最前面，而应该作为“本轮附加信息”后移。

### 4.3 历史对话层

这类内容随着轮次增加一定会膨胀，所以必须有收纳机制。

策略不是一味保留，也不是一刀切删除，而是：

- 最近的内容保持原样
- 更早的内容在必要时压缩成较短摘要

这样才能在“记得住”和“撑不爆”之间找到平衡。

## 5. 这次具体做了什么方向的调整

从产品和策略角度看，这次主要做了四件事。

### 5.1 让稳定规则尽量回到稳定位置

之前有些本来应该稳定存在的规则，被拆得比较散，甚至和临时信息混在一起。

这次把它们重新收敛，让模型更容易形成固定理解。

最典型的是思考动作的规则：保留约束，但减少重复和碎片化。

### 5.2 把高波动信息后移

本轮会变化的信息，不再尽量往最前面塞，而是后移到更靠近“本轮输入说明”的位置。

这样做的好处是：

- 规则前缀更稳定
- 临时状态还在，不会丢
- 模型能区分“长期规则”和“本轮状态”

### 5.3 让系统提示不要随机波动

如果每次人格、风格、背景故事有时出现有时缺失，模型会很难形成稳定行为。

这次策略上明确选择了：

- 前缀先求稳定
- 不再依赖随机缺段来“增加自由度”

自由度可以通过后面的动态上下文和对话本身获得，而不应该通过破坏骨架稳定来换。

### 5.4 给长对话准备“压缩而不是遗忘”的路径

这次没有选择复杂的大型总结流程，而是先给系统一个稳定、低风险的收纳思路：

- 内容太多时，不是直接忘掉
- 而是把更早的内容折成简短摘要继续带着走

这个策略的重点不是“摘要多聪明”，而是“上下文不会突然断层”。

## 6. 为什么没有直接上更激进的方案

理论上，可以做得更激进，比如：

- 把长对话定期做高质量总结
- 每隔几轮就做一次深度压缩
- 用更复杂的方式区分永久记忆、短期记忆和临时心流

但这次没有一步走那么远，原因很现实：

- 先把结构整理清楚，收益最大
- 一上来引入太多新机制，调试成本会很高
- 如果基础分层没做好，后面的高级压缩只会把混乱再次包装起来

所以这次的选择是：

- 先把输入结构变干净
- 再把动态内容位置放对
- 最后再给历史内容一个可持续的收纳方式

这是一个更稳的顺序。

## 7. 这次策略对系统会带来什么影响

### 7.1 对缓存

这是最直接的收益点。

当输入前缀更稳定以后：

- 缓存命中条件更容易满足
- 重复轮次之间的公共部分更多
- 延迟和成本会更可控

### 7.2 对行为一致性

模型会更容易“记住自己该怎么说话、怎么行动”。

这不是因为模型突然更强了，而是因为输入结构变得更像一套稳定制度，而不是一堆临时拼接的信息。

### 7.3 对长对话质量

长对话里最怕的不是上下文变少，而是“前面突然被忘掉，而且没有任何痕迹”。

现在的方向是让旧内容被压缩成可追踪的摘要，而不是直接消失。

这会让长对话在连续性上更自然一些。

## 8. 这次还没有完全解决的地方

目前仍然有一些系统状态不是完全静态的。

也就是说，这次已经把最大的扰动源整理掉了，但还没有做到“全系统前缀完全稳定”。

从整体上看，接下来还有两个方向值得继续推进：

- 继续把少量仍会影响前缀的动态说明后移
- 观察长对话里压缩摘要的实际表现，再决定是否要升级成更强的总结机制

换句话说，这次不是终局，而是先把地基打平。

## 9. 关于 payload 的策略结论

对于上下文长度问题，这次的结论不是“到阈值就删”，也不是“到阈值就做复杂总结”，而是一个更稳的中间路线：

1. 先尽量减少不必要的前缀波动
2. 再把高频变化的信息放到更靠后的位置
3. 对动态信息本身做一定节制
4. 只有在真的变长之后，才对更早的内容做压缩收纳

这意味着，payload 的管理不是一个单点动作，而是一整套顺序：

- 先整理结构
- 再控制增长
- 最后处理溢出

这比单纯的“阈值触发压缩”更可靠。

## 10. 为什么这份策略是合理的

如果把整个系统看成一个长期运行的对话体，那么最重要的不是一轮里塞了多少信息，而是：

- 模型能不能先看到稳定骨架
- 临时状态是不是在合适的位置出现
- 历史内容是不是能被持续整理，而不是越堆越乱

这次改造，本质上是在做“信息秩序重排”。

它不是追求更华丽的功能，而是在建立一种更可持续的输入结构。

## 11. 后续建议

后续建议分两步走。

### 第一步：继续稳定前缀

目标是尽可能把还会影响前缀稳定性的动态内容继续后移，进一步提升缓存命中率。

### 第二步：观察压缩效果

先看一段时间实际对话表现：

- 是否明显减少前缀波动
- 是否长对话更稳定
- 是否摘要后的上下文仍能维持连续感

如果这些观察结果是正面的，再决定要不要引入更高级的总结机制。

## 12. 总结

这次策略的核心，不是“删东西”，而是“把东西放到该放的位置”。

一句话总结就是：

- 固定规则，尽量固定
- 临时状态，尽量后移
- 历史对话，必要时压缩

这样一来，系统会更像一个有秩序的长期对话体，而不是每轮都临时拼装的大杂烩。
